# AI 生成代码采纳率归因分析方案

### 1. 方案目标

本方案用于解释 AI 首轮生成代码在最终交付过程中被人工修改、删除或补充的原因，并将问题归因到研发链路中的具体阶段和失败机制。

它关注的不是"最终代码有没有变化"，而是：

1. final 相对 one-shot 到底改变了什么行为；
2. 这些改变是否属于有效逻辑修正，而不是测试、文档、自动生成代码、格式或注释噪声；
3. 如果是有效修正，AI 首轮为什么没有生成正确代码；
4. 问题最早出现在哪个阶段；
5. 该问题是知识缺失、知识传递损耗、执行写入损耗、模型推理偏差、门禁漏检，还是澄清不足；
6. 当前证据是否足以支撑该归因结论。

最终目标是把"采纳率下降"拆解成可复核、可统计、可优化的问题画像，而不是只输出一个主观标签。

### 2. 核心思想

#### 2.1 代码差异只是症状，不是归因结论

one-shot 到 final 的代码变化只说明"最终交付版本与 AI 首轮结果不同"。这只是问题入口，不能直接推出 AI 有问题，也不能直接推出某个阶段有问题。

同一段变化可能对应多种解释：

| 代码变化 | 可能解释 |
| --- | --- |
| 人工补充参数 | AI 漏实现，也可能是设计文档漏写接口契约 |
| 人工删除分支 | AI 生成了多余逻辑，也可能是需求后续变化 |
| 人工补充判空 | 任务计划漏了异常分支，也可能是设计阶段没有边界条件 |
| 人工调整 import/注释 | 可能只是无归因价值的代码噪声 |

因此，归因必须从代码症状出发，继续追问：当 AI 生成 one-shot 时，正确约束是否已经存在、是否被传递、是否被写入产物、AI 是否遵循了产物。

#### 2.2 归因是证据驱动的假设检验

归因过程不是先选标签，而是对多个可能原因提出假设，再用代码、产物、执行轨迹和门禁证据逐一验证或排除。

每个归因结论都必须回答：

```Plain Text
现象是什么？
应遵循的约束是什么？
约束在当时是否存在？
约束是否进入了中间产物？
AI 首轮代码是否遵循该产物？
人工 final 是否把代码修回了该约束？
如果没有修回约束，是否存在需求后变更、偏好调整或非目标链路？
```

只有当证据闭环成立时，才输出 confirmed 或 probable。证据不足时输出 insufficient_evidence；不属于本次目标链路或需求后变更时输出 not_attributable。

#### 2.3 产物缺陷优先于 AI 执行偏差

在 N5/N4 等 code ↔ artifact 对比场景中，不能只看 before-side 与产物是否一致。

核心判断矩阵如下：

| before-side vs artifact | after-side vs artifact | 判断 |
| --- | --- | --- |
| consistent | consistent | AI 和人工最终代码都遵循产物，通常不是该产物层首因 |
| consistent | inconsistent | AI 忠实执行了产物，人工 final 偏离产物，优先怀疑产物滞后、产物错误或需求后变更 |
| inconsistent | consistent | 人工把代码修回产物约束，可稳定指向 AI 执行偏差 |
| inconsistent | inconsistent | 复合场景，必须先判断 artifact 是否可信，不能直接归因 AI |

这个原则避免把"AI 忠实执行了错误产物"的问题误判为 AI 代码生成问题。

#### 2.4 归因粒度是 Change Intent，不是单个 Hunk

一个业务修正常常跨多个文件、多个仓库和多个 hunk。例如接口字段调整可能同时涉及 IDL、DTO、服务实现、Controller、调用方。

如果逐 hunk 归因，会把同一个问题拆成多个重复结论，也容易产生前后不一致。因此方案采用：

```Plain Text
Raw Diff Hunk -> Change Slice -> Change Intent -> Finding -> Problem Type -> Root Cause
```

Hunk 只保留代码事实；真正的问题判断、证据链、根因和置信度只落在 intent 级。

#### 2.5 先判断是否值得归因

不是所有代码变化都应该进入归因。归因前先排除不影响业务逻辑或无法代表 AI 代码质量的噪声：

| 排除对象 | 原因 |
| --- | --- |
| 文档、报告、静态 HTML | 不代表运行时代码逻辑 |
| 测试文件、fixture、mock | 不进入主业务代码采纳率问题 |
| 自动生成代码 | 人工修改可能来自 codegen，不代表 AI 生成逻辑 |
| 纯配置值、版本号 | 多数不代表业务逻辑缺陷 |
| 纯空白、纯注释、纯 import | 不影响行为 |

排除规则的目的不是"让指标好看"，而是保证采纳率和归因只关注对最终交付逻辑有意义的变化。

### 3. 归因考虑的核心维度

#### 3.1 代码事实维度

代码事实回答"实际发生了什么变化"。

| 维度 | 关注点 |
| --- | --- |
| 版本边界 | base、one-shot、final 是否明确 |
| 变更方向 | 新增、删除、修改、重构 |
| 影响范围 | 单文件、跨文件、跨仓库 |
| 行为语义 | 修正了什么业务行为、接口契约或边界条件 |
| 有效性 | 是否排除文档、测试、自动生成、纯注释等噪声 |
| 指标影响 | one-shot_lines、removed_lines、retained_lines、final_lines |

代码事实是归因的锚点。没有代码事实，不能做归因；但只有代码事实，也不能直接下结论。

#### 3.2 改动性质维度

不同类型的代码修正，需要进入不同的归因路径。

| 改动性质 | 典型含义 | 归因重点 |
| --- | --- | --- |
| functional_logic | 业务逻辑、条件分支、算法 | 需求、设计项、任务步骤是否完整正确 |
| interface_contract | 接口签名、参数、返回值、DTO、IDL | 接口契约是否遗漏或不一致 |
| data_model | 表、字段、索引、缓存键 | 数据模型设计是否完整 |
| non_func_constraint | 日志、异常、防御、事务、幂等 | 约束是否存在并被引用 |
| architecture_constraint | 分层、依赖方向、模块边界 | 架构约束是否存在并被遵循 |
| stability_measure | 限流、降级、灰度、回滚 | 稳定性方案是否遗漏 |
| coding_style | import、格式、命名、注释 | 多数短路到编码规范或直接排除 |
| structural_refactor | 抽方法、复用、移动代码 | 复用模式或约束是否存在 |

改动性质决定后续应该检查哪些产物，而不是对所有产物做泛泛扫描。

#### 3.3 阶段维度

归因沿Agent研发链路自下而上穿透：

| 阶段 | 核心问题 |
| --- | --- |
| N5 编码计划 | Task 是否完整承接设计，AI 是否遵循任务 |
| N4 技术方案 | 设计项、接口、存储、稳定性、约束是否完整正确 |
| N3 需求澄清 | 用户故事、AC、范围、流程、状态是否完整清晰 |
| N2 现状梳理 | 当前系统、GAP、影响范围、领域知识是否识别充分 |
| N1 项目初始化 | 业务领域、仓库范围、原始 PRD 功能是否界定正确 |

阶段维度回答"问题最早在哪里产生"。如果 N5 只是忠实继承了 N4 的错误，首因应归到 N4，而不是 N5。

#### 3.4 缺陷形态维度

缺陷形态描述"对象以什么方式出错"。

| 缺陷形态 | 含义 |
| --- | --- |
| existence | 应存在的条目完全缺失 |
| completeness | 条目存在但覆盖不全 |
| correctness | 条目存在但内容错误 |
| quality | 条目存在但模糊、不可执行、不可测试 |
| deviation | AI 代码偏离了已有正确产物 |

缺陷形态与阶段对象组合后，形成问题类型假设。例如：

```Plain Text
N4 + design_items + existence = 设计项遗漏
N4 + interface_def + correctness = 接口字段不一致
N5 + ai_deviation + deviation = 编码偏离计划
N3 + ac_testability + quality = AC 不可测试
```

#### 3.5 产物对象维度

每个阶段只检查与当前改动相关的产物对象。

| 阶段 | 典型对象 |
| --- | --- |
| N5 | Task、Instruction、覆盖矩阵、constraint 引用 |
| N4 | D-xx、DEC-xx、接口定义、存储方案、稳定性、约束 |
| N3 | Scope、操作流、状态机、用户故事、AC |
| N2 | 入口清单、核心流程、GAP、影响范围、行为差异、知识条目 |
| N1 | 业务领域、仓库范围、原始需求、领域分类 |

产物对象维度避免"发现代码改了接口，却去检查无关的稳定性章节"这类误归因。

#### 3.6 根因机制维度

问题类型说明"哪里错了"，根因机制说明"为什么会错"。

| 根因 | 假设 |
| --- | --- |
| R1a 知识源头缺失 | 完成任务所需知识在知识源中不存在 |
| R1b 知识传递损耗 | 知识源存在，但未被检索、加载或注入到产物 |
| R2 执行损耗 | 上游或中间推理中已有信息，但最终产物写入时丢失或改错 |
| R3 模型推理 | 信息和知识充分，但模型转换、推理或生成时出错 |
| R4 门禁漏检 | 问题本应被 review/gate 发现，但未被拦截 |
| R5 澄清交互不充分 | 需要澄清的问题没有被提出或没有充分追问 |

同一个问题类型可能对应多个根因候选，必须逐个核验，而不是看到 P-code 后直接映射 R-code。

#### 3.7 证据维度

归因需要多源证据：

| 证据类型 | 作用 |
| --- | --- |
| 代码证据 | 证明 before/final 的实际变化 |
| 产物证据 | 证明需求、设计、任务、约束是否存在 |
| 执行轨迹证据 | 证明知识是否被检索、产物是否被写入、信息是否在中间环节出现过 |
| Gate 证据 | 证明门禁是否存在、是否执行、是否漏检 |
| 知识证据 | 证明知识源是否存在或缺失 |
| 澄清证据 | 证明问题是否应澄清、是否被追问 |
| 手工补充证据 | 在自动证据不足时作为低优先级辅助来源 |

结论必须由 evidence_items 和 evidence_refs 形成闭环，不能只靠自然语言解释。

#### 3.8 置信度与归因状态维度

归因结论分为四类：

| 状态 | 含义 | 是否进入正式统计 |
| --- | --- | --- |
| confirmed | 证据闭环完整 | 是 |
| probable | 主证据成立，但有非关键证据缺口 | 是，但报告标记缺口 |
| insufficient_evidence | 缺关键证据或存在同等解释 | 否 |
| not_attributable | 需求后变更、偏好调整或非目标链路 | 否 |

置信度不是模型自信，而是证据完整度、根因唯一性和可复核性的综合结果。

### 4. 问题假设的构造方法

#### 4.1 从代码症状提出初始假设

每个 Change Intent 先从 final 相对 one-shot 的变化提出"修正意图"：

```Plain Text
人工 final 做了什么？
它修正了 AI 的什么行为？
这个行为是否可由需求、设计、任务或约束解释？
```

初始假设包括：

| 假设 | 问题 |
| --- | --- |
| H0 非归因对象 | 这是文档、测试、生成代码、格式、注释或无逻辑影响改动吗 |
| H1 需求后变更 | final 是否体现 one-shot 之后才出现的新需求 |
| H2 AI 执行偏差 | 正确产物存在，AI 是否没有遵循 |
| H3 编码计划缺陷 | 设计正确，但 tasks.md 是否拆解遗漏或写错 |
| H4 技术方案缺陷 | 需求正确，但 design/interface/storage/stability 是否遗漏或错误 |
| H5 需求澄清缺陷 | 原始需求或现状正确，但 requirement.md/AC 是否缺失、模糊或错误 |
| H6 现状梳理缺陷 | 系统现状、GAP、影响范围或知识加载是否不充分 |
| H7 项目初始化缺陷 | 业务领域、仓库范围或原始 PRD 功能是否识别错误 |

H0/H1 优先用于防止强行归因；H2-H7 用于定位首因层。

#### 4.2 按改动性质选择检查路径

不是所有问题都走完整 N5→N1。

| 改动性质 | 假设路径 |
| --- | --- |
| 功能、接口、数据、稳定性 | 正常穿透：N5→N4→N3→N2→N1 |
| 非功能、架构、结构性重构 | 沿 C 项约束链穿透：Task/D-xx 是否引用 constraint，constraint 是否来自知识 |
| 编码风格 | 短路：优先判断是否无归因价值，或只检查编码规范约束 |

这样可以减少无关假设，避免让所有代码改动都归到需求或设计层。

#### 4.3 每层先问产物是否正确，再问 AI 是否遵循

在每一层，问题假设分两类：

1. 产物缺陷假设：当前阶段产物本身有遗漏、错误、模糊或引用缺失；
2. 执行偏差假设：当前阶段产物正确，但 AI 代码或下游产物没有遵循。

判断顺序为：

```Plain Text
先检查产物完整性与正确性
再检查 before-side 是否遵循产物
最后检查 after-side 是否证明人工修正方向
```

如果产物本身错误，AI 遵循错误产物不应被判为主因。

#### 4.4 用矩阵形成问题类型假设

问题类型不是 LLM 随机选择，而是由三元组构成：

```Plain Text
stage + artifact_aspect + defect_form
```

示例：

| stage | artifact_aspect | defect_form | 问题假设 |
| --- | --- | --- | --- |
| N5 | reference_mapping | existence | Task 未关联上游 D-xx/C-xx |
| N5 | task_step_coverage | completeness | 任务步骤覆盖不全 |
| N4 | design_items | existence | 设计项遗漏 |
| N4 | interface_def | correctness | 接口字段不一致 |
| N3 | ac_list | completeness | AC 列表覆盖不全 |
| N2 | impact_scope | completeness | 影响范围遗漏 |
| N1 | repo_scope | correctness | 仓库范围错误 |

这种矩阵化方式保证问题类型可枚举、可统计、可验证。

#### 4.5 再对根因机制提出假设

确定"哪里错了"后，再判断"为什么错"。

根因假设按证据链核验：

| 根因假设 | 需要证明 |
| --- | --- |
| R1a | 该阶段需要某知识，但知识源确实缺失 |
| R1b | 知识源存在，但执行轨迹未检索、未加载或产物未引用 |
| R2 | 上游或中间推理已有正确内容，但最终产物写入时丢失 |
| R3 | 输入和知识都充分，产物仍错误，且 R1/R2 被排除 |
| R4 | 有明确 Gate 应检查该问题，但 Gate 通过或缺失 |
| R5 | 该问题需要澄清，但澄清日志缺少关键提问 |

如果多个根因都可能成立，优先选择证据链最直接、最能解释 after-side 修正的根因；其他可作为附加标签或竞争候选记录。

#### 4.6 证据不足时不强行归因

不能证明时，结论应降级：

| 场景 | 处理 |
| --- | --- |
| 缺少关键产物 | 标记 evidence_gap，降低置信度 |
| 缺少 execution_trace | 不能确认 R1b/R2，只能 probable 或 insufficient |
| before/after 都不符合产物 | 先判断产物可信度，否则不能判 AI 偏差 |
| final 体现新需求但无正式证据 | 可标记 possible post-one-shot change，但不能高置信 |
| 与需求链路无关 | not_attributable |

归因宁可暴露证据缺口，也不能生成看似完整但不可复核的结论。

### 5. 归因判断流程

#### 5.1 总流程

```Plain Text
1. 明确 base / one-shot / final 三个版本
2. 生成并校验三段 diff
3. 排除无逻辑影响代码
4. 将有效代码变化聚合为 Change Intent
5. 判断 change_nature
6. 提取 before-side 与 after-side 行为差异
7. 查找 one-shot 时应遵循的约束
8. 按阶段提出并验证问题假设
9. 用矩阵派生问题类型
10. 对 R1a/R1b/R2/R3/R4/R5 做根因核验
11. 汇总证据状态和置信度
12. 输出归因结论、证据缺口和优化建议
```

#### 5.2 单个 Intent 的判断过程

```Plain Text
Change Intent:
  final 修改了什么？
  是否排除噪声？
  属于哪类 change_nature？

约束定位:
  N5 是否有对应 Task？
  N4 是否有 D-xx/interface/storage/stability/constraint？
  N3 是否有 US/AC/scope/flow/state？
  N2 是否有 current-state/knowledge/evidence？
  N1 是否有领域、仓库、原始需求定义？

首因判断:
  哪一层最早出现缺失/错误/模糊/偏离？

问题分类:
  stage + artifact_aspect + defect_form -> problem_type

根因核验:
  R1a/R1b/R2/R3/R4/R5 哪个证据闭环成立？

状态输出:
  confirmed / probable / insufficient_evidence / not_attributable
```

### 6. 典型问题假设示例

#### 6.1 AI 编码偏离计划

假设：

```Plain Text
tasks.md 已正确要求实现某逻辑，但 AI one-shot 没按任务实现，final 人工修回任务约束。
```

需要证据：

1. Task 中存在明确约束；
2. before-side 与 Task 不一致；
3. after-side 与 Task 一致；
4. Task 本身能追溯到正确的上游 D-xx 或 C-xx；
5. 若要判 R3，还需排除知识缺失和写入损耗。

#### 6.2 设计项遗漏

假设：

```Plain Text
需求或 AC 已要求某行为，但 design.md 没有设计对应 D-xx，导致 tasks.md 和 one-shot 都缺失该行为，final 人工补回。
```

需要证据：

1. requirement.md / AC 中存在该约束；
2. design.md 缺少对应设计项；
3. tasks.md 忠实继承了缺失设计；
4. before-side 缺失该逻辑；
5. after-side 补回该逻辑。

#### 6.3 接口字段不一致

假设：

```Plain Text
接口契约中字段、类型、参数传递不一致，导致 AI 生成代码无法满足最终调用链。
```

需要证据：

1. final 修改体现接口字段或参数变化；
2. design-interface 或 D-xx 中接口定义缺失/错误；
3. 相关实现、调用方、DTO/IDL 共同变化；
4. after-side 修正形成一致契约。

#### 6.4 知识传递损耗

假设：

```Plain Text
知识库存在相关规范或模式，但执行时没有召回、没有加载，或加载后没有写入阶段产物。
```

需要证据：

1. 知识源存在该条目；
2. execution_trace 未检索或未注入；
3. 产物缺少该知识对应约束；
4. final 人工修正体现该知识确实需要。

#### 6.5 门禁漏检

假设：

```Plain Text
问题已经出现在某阶段产物中，并且对应 Gate 本应检查该类问题，但 Gate 未发现。
```

需要证据：

1. Gate 有对应检查项，或按职责应检查；
2. 缺陷对 Gate 可见；
3. Gate 结果通过、缺失或未覆盖；
4. 该问题最终通过人工代码修正暴露。

### 7. 输出结论结构

每个 intent 的归因结论应包含：

| 字段 | 含义 |
| --- | --- |
| change_summary | final 相对 one-shot 的行为变化 |
| change_nature | 改动性质 |
| first_cause_stage | 首因阶段 |
| findings[] | 发现的问题事实 |
| problem_types[] | 矩阵派生的问题类型 |
| root_causes[] | 根因机制及候选核验 |
| evidence_chain[] | 从代码到产物再到执行轨迹的证据链 |
| evidence_items[] | 可复核证据池 |
| attribution_status | 证据状态 |
| missing_evidence[] | 缺失证据 |
| not_attributable_reason | 不可归因原因 |

最终报告不只展示"问题是什么"，还要展示"为什么这样判"和"证据缺在哪里"。

### 8. 方案边界

本归因方案不做以下假设：

1. 不假设 final 修改一定代表 AI 错误；
2. 不假设 before-side 与产物不一致就一定是 AI 主因；
3. 不假设所有变更都应该进入采纳率和归因统计；
4. 不用 agcr 观测值代替 diff 独立计算；
5. 不在证据不足时强行输出高置信根因；
6. 不将阶段名直接等同于责任方；
7. 不将问题类型直接等同于根因。

本方案的核心取舍是：以可复核证据为优先，允许暴露证据不足，不追求每个代码变化都被强行解释。
