# Agent代码采纳率归因分析方案

### 一、方案概述

#### 1.1 背景与目标

**代码采纳率**（AGCR，AI-Generated Code Ratio）是衡量Agent产出质量的核心指标，反映 AI 生成代码被 RD 实际采纳的比例。在实际工程实践中，同一个低采纳率问题往往有多层成因：表象层可能是集成测试未通过，中间层是详设 Spec 不够细，源头层则是需求 Spec 本身存在歧义。单层归因只能定位"最近的原因"，无法指导根本性改进。

本方案旨在建立一套**系统化的采纳率归因分析框架**，实现：

1. **辅助分析**：帮助 RD/PM/QA 快速定位采纳率低的根因，支持多层深推，指导改进方向
2. **工具评测**：通过归因结果对Agent各 Skill 进行量化评测，支持 Skill 自迭代
3. **闭环驱动**：提供事前预防、事中检测、事后分析三位一体的归因闭环，驱动 Spec 三阶段迭代落地
4. **可扩展性**：框架设计支持新增归因维度和问题类型，随流程演进持续完善
#### 1.2 分析时机

**事中检测（In-flight）**：包含两类触发机制。一是**阶段完成自检**：Skill 执行过程中在各关键节点（N3/N4/N6/N8/QA3）完成后，对产出物质量进行自检，输出阶段评测打分；二是**异常信号捕捉**：通过 hook 持续监听执行过程中的异常行为信号，包括人工多次纠偏（同一阶段修改次数超阈值）、超长时间运行（单阶段耗时超阈值）等，触发后基于当前上下文进行实时问题诊断（分析当前agent是否有问题）。

**事后分析（Post-flight）**：需求交付后，基于完整 context（需求记录、AI-trace、代码 diff）进行深度归因复盘，输出归因分布报告。

#### 1.3 核心分析逻辑

采纳率归因遵循以下分析路径：输入最新版代码版本和one-shot提交代码版本，比较两个版本代码文件之间的Diff Hunk[^5d9107f5-bf3e-41be-8bb2-6833b5ba5648]（变异块/变更块，后文简称 Hunk），以 Hunk 作为最小归因单元逐步进行归因分析。

### 二、关键定义

#### 2.1 **Hunk定义**

##### **2.1.1 Hunk vs Commit vs 功能调整（Feature Change）的关系**

<!-- 代码块 -->
```Plain Text
需求（Requirement）
  └── 功能调整（Feature Change）—— 语义层：一个独立的业务意图变更
        └── Commit（s）—— 版本控制层：一次代码提交，可能包含多个文件
              └── Hunk（s）—— 归因分析层：一个原子性的语义修改片段
```

| **维度** | **Hunk** | **Commit** | **功能调整（Feature Change）** |
| --- | --- | --- | --- |
| **本质** | 一段**语义连贯**的代码修改片段 | 一次 Git 提交操作 | 一个**完整业务意图**的实现 |
| **划分依据** | 语义边界（LLM 判断修改意图是否同一） | RD 手动提交时机（可能随意） | 业务功能边界 |
| **粒度** | 最小，通常 5~50 行 | 中等，可能包含多个不相关改动 | 最大，可能跨多个文件和 commit |
| **与归因的关系** | 归因的**最小分析单元** | 归因的**辅助定位线索** | 归因的**汇总报告单元** |

<details>
<summary>**为什么不用 Commit 作为归因单元？**</summary>

Commit 是 RD 的**版本控制行为**，而不是**语义分析单元**。实际工程中存在几个问题：

第一，**一个 Commit 可能混杂多种修改意图**。RD 可能在一次提交中既修了一个逻辑 bug、又顺手调了命名风格、还补了个 null check。这三处改动的归因方向完全不同（分别对应 P3 设计缺陷、P4 规范违反、P5 健壮性缺失），如果以 Commit 为单位只能给出一个模糊的归因。

第二，**一个功能调整可能分散在多个 Commit 中**。RD 可能先提交主逻辑、再提交异常处理、最后提交单测修复。这些 Commit 服务于同一个功能调整，归因结论应该是统一的。

第三，**Commit 的粒度因人而异**。有人习惯频繁小 Commit，有人习惯一次大 Commit，这个变量不应该影响归因分析的质量。

</details>

##### 2.1.2 **Hunk**[^d7a527db-fe62-4b9d-b637-88f4cb861517]** 的划分规则**

Hunk 的划分不是按 Git 的物理 diff hunk（`@@` 行分隔），而是按**语义边界**重新切分：

<!-- 代码块 -->
```Plain Text
/**
 * Hunk 划分规则（Phase 1 Step 1）
 * 
 * 输入：RD 对 AI 生成代码的完整 Git Diff（可能跨多个 commit）
 * 输出：语义独立的 Hunk 列表
 */
interface HunkSegmentationRules {
  /**
   * 合并条件：以下情况的多个物理 diff 片段应合并为一个 Hunk
   */
  mergeWhen: [
    '同一方法内、服务于同一修改意图的连续/相邻改动',
    '跨文件但服务于同一接口变更（如改了接口定义 + 改了调用方）',
    '一处逻辑修改 + 其必要的配套修改（如改了枚举值 + 改了 switch case）',
  ];

  /**
   * 拆分条件：以下情况即使在同一物理 diff hunk 中也应拆分为多个 Hunk
   */
  splitWhen: [
    '同一文件中存在多个不相关的修改意图',
    '一段改动既包含逻辑修复又包含风格调整',
    '一段改动中部分是功能新增、部分是 bug 修复',
  ];

  /**
   * 排除条件：以下改动不参与归因分析
   */
  excludeWhen: [
    '纯空白/格式化变更（IDE 自动格式化）',
    'import 语句的自动整理',
    '自动生成代码的 regenerate（如 protobuf stub）',
    'RD 明确标记为 "个人偏好" 的修改（从采纳率计算中剔除）',
  ];
}
```

**具体示例**

<!-- 假设一个需求："营销活动增加互斥规则校验"，RD 提交了 3 个 Commit： -->
```Plain Text
Commit A: "fix: 修复互斥规则校验逻辑"
  ├── Hunk 1: 修正了 checkMutualExclusion() 的判断条件 [功能逻辑错误] → 归因 P3 设计缺陷

Commit B: "feat: 补充互斥规则的异常处理"  
  ├── Hunk 4: 增加了 ConcurrentModificationException 处理 [缺少防御性代码] → 归因 P5
```

<!-- 更新后的 Hunk 结构（补充与 Commit / Feature Change 的关联） -->
```Plain Text
interface Hunk {
  // ... 之前定义的字段 ...

  // ===== 与 Commit 的关系 =====
  sourceCommits: string[];            // 来源 commit hash 列表（一个 Hunk 可能跨 commit）
  commitMessage?: string;             // 相关 commit message（辅助意图识别）
  taskRef?: string;                   // commit message 中引用的 task 编号（如 "Task-003"）

  // ===== 与 Feature Change 的关系 =====
  featureChangeId?: string;           // 所属功能调整 ID（多个 Hunk 共享）
  featureChangeDescription?: string;  // 功能调整描述
  
  // ===== Hunk 间关系 =====
  relatedHunks?: string[];            // 语义关联的其他 Hunk ID（如接口定义改了 + 调用方也改了）
  isCompanionOf?: string;             // 若本 Hunk 是另一个 Hunk 的配套修改，指向主 Hunk
}

/**
 * 功能调整（Feature Change）：归因报告的业务聚合单元
 * 
 * 一个 Feature Change 是 RD 为完成一个独立业务意图所做的所有代码修改的集合。
 * 它是面向 PM/RD 的归因报告粒度，而 Hunk 是面向 Skill 改进的技术分析粒度。
 */
interface FeatureChange {
  featureChangeId: string;
  description: string;                   // 业务意图描述
  relatedUserStory?: string;             // 关联的 US 编号（requirement.md 中的）
  relatedDesignItem?: string;            // 关联的 D-xx 编号（design.md 中的）
  
  commits: string[];                     // 包含的 commit 列表
  hunks: string[];                       // 包含的 Hunk ID 列表
  
  // 汇总归因（从 Hunk 级别聚合）
  aggregatedAttribution: {
    primaryCategory: ProblemCategory;    // 主要问题类型（最高频）
    primaryLayer: AttributionLayer;      // 主要首因层
    allCategories: Record<ProblemCategory, number>;  // 所有问题类型分布
    dominantPath: string;                // 最高频传导链路
    improvementPriority: 'HIGH' | 'MEDIUM' | 'LOW';  // 改进优先级
  };
}
```

##### 2.1.3 **Hunk问题类型**

| **人工修改的性质** | **最可能的问题文件** | **归因类** |
| --- | --- | --- |
| 功能逻辑完全错误/多余/缺失 | requirement.md（需求没写清楚）或 design.md（方案设计错误） | P1 / P3 |
| 功能正确但与现有代码行为冲突 | current-state.md / evidence.md（现状描述不完整） | P2 |
| 功能正确但违反编码规范/中间件用法 | knowledge.md / constraint-check.md（约束未收录或不够具体） | P4 |
| 功能正确但缺少防御性代码（null check、异常处理） | design.md 的行为契约不完整 + constraint-check.md 缺少反模式 | P5 |
| 实现方式与团队风格不一致（命名、分层、模式） | knowledge.md 中 yx-domain-knowledge 的团队约定不全 | P4 |
| 接口签名/参数/返回值不对 | design.md §5 接口定义不精确 | P3 |

总结来说，**最高频导致采纳率低的文件质量问题**大概率集中在三个地方：

1. **design.md 的行为契约精度不够**——"做什么"写清楚了，但"边界怎么处理"没写清楚
2. **current-state.md 的覆盖面不够**——只描述了主流程，遗漏了被改动代码的隐式依赖关系
3. **knowledge.md/constraint-check.md 的可执行性不够**——约束太抽象，没有具体到"哪个场景、哪个字段、用什么方式"

### 三、问题分类体系

#### 3.1 三维度分类框架

采纳率损失的根因可从三个正交维度进行归类，每个维度回答一个不同层面的问题：

| 维度 | 核心问题 | 关注点 | 对应分类 |
| --- | --- | --- | --- |
| D1 产出物质量 | 某阶段的产出物本身质量是否达标？ | 各阶段产出物（Spec、current-state、design、code）的内在质量 | P0~P5（可扩展） |
| D2 过程传递 | 跨阶段信息传递是否完整无损？ | 阶段间上下文传递、SubAgent 结果聚合、产出物路径衔接 | P6 |
| D3 执行遵循 | 有正确产出物但执行时是否遵循了？ | PCR 流程遵循度——产出物质量 OK 但后续阶段未遵守或跳过 | PCR 偏差（详见第四章 4.1 节） |

**三维度的关系与使用顺序**：归因时按 D1→D2→D3 的顺序进行——先通过逆向归因定位首因阶段（D1 产出物质量），确定"哪个阶段的产出物出了问题"；在首因阶段确定后，进一步判断该问题是由 D2（过程传递）导致的信息断层引起，还是由 D3（执行遵循）导致的流程跳过引起。三个维度互相正交：D1 关注"做得好不好"，D2 关注"传得全不全"，D3 关注"有没有做"。

##### 3.1.1 三维度归因决策流程

综合三个维度，完整的归因决策流程为：

<!-- 代码块 -->
```Plain Text
Hunk 被修改/拒绝
  │
  ├─► Step 1: 逆向回溯定位问题表现层
  │     └─ 从最终代码沿 pipeline 逆向回溯，找到第一个"输出与预期不符"的阶段
  │           └─ 输出：问题表现层阶段（N1~N6）
  │

        │

        │     对比理想态，该阶段产出物是否存在质量不足
        │     ├─ 是 → 标记 D1 产出物质量问题
        │     │     └─ 归入 P0~P5 具体分类，或自行定义新类型
        │     └─ 否 → 跳过
        │
        ├─ 2b: 上游信息在传递中丢失/失真？
        │     检查上游阶段产出物是否存在且质量 OK，但该层 Context 中缺失
        │     ├─ 是 → 标记 D2 过程传递问题（P6）
        │     │     └─ 定位哪两个阶段之间断层 + 丢失的具体信息
        │     └─ 否 → 跳过
        │
```

**说明**：归因流程分两步——Step 1 通过逆向回溯定位"问题表现层"（第一个输出与预期不符的阶段），Step 2 在该阶段上逐一检查三个维度。三个维度之间不互斥，一个阶段可能同时存在多个维度的问题（例如：上游信息传递丢失导致输入不全 [D2]，同时该阶段对已有的输入也未充分利用 [D1]，且流程约束被跳过 [D3]）。每个命中的维度独立输出归因结论和改进方向。

**实现分期**：第一版实现聚焦 D1（产出物质量）维度的归因——仅依赖各阶段产出物（requirement.md、current-state.md、design.md、tasks.md 等静态文件）即可完成归因判定。D2（过程传递）和 D3（执行遵循）的归因需要依赖执行过程中的 trace 日志（如 Context 传递记录、Skill 调用链、门禁确认记录等运行时数据），将在后续迭代中实现。第一版中，决策树定位首因层后默认归入 D1，输出产出物质量分析结论。

#### 3.2 维度一：产出物质量（D1）

产出物质量维度关注各阶段输出产物的内在质量。当某阶段的产出物本身存在缺陷（信息不完整、逻辑错误、覆盖不全），将直接导致后续阶段基于有缺陷的输入生成低质量代码。这是最核心的归因维度，预定义 P0~P5 共 6 个一级分类，归因时若不匹配已有类型可自行定义新分类。

##### 3.2.1 分类设计原则

产出物质量分类遵循 **MECE 原则**：

4. **互斥性**：每个被修改/拒绝的代码片段只归属一个一级类型
5. **完备性**：所有采纳率损失场景均可被覆盖
6. **可操作性**：每个分类对应明确的改进方向和责任 Skill
7. **可扩展性**：归因时若问题不匹配 P0~P5 已有分类，模型可自行定义新的问题类型并给出明确定义和典型表现
##### 3.2.2 一级分类总览（6 类 + 可扩展）

| 分类 | 名称 | 常见关联阶段 | 典型表现 |
| --- | --- | --- | --- |
| P0 | 项目初始化与外部输入问题类 | N1 项目初始化 | PRD 需求遗漏、与本需求无关的代码引入、知识库未收录仓库 |
| P1 | 需求理解偏差类 | N3 需求澄清 | 需求描述模糊、验收标准缺失、Spec 质量不足 |
| P2 | 现状基线偏差类 | N2 现状梳理 | 代码库理解偏差，生成代码与现有架构不符 |
| P3 | 技术方案设计缺陷类 | N4 技术方案 | 接口定义不完整、推导链断裂、设计质检漏检 |
| P4 | 代码规范与架构约束违反类 | N6 代码生成 | 架构分层违反、中间件使用不当、风格不一致 |
| P5 | 代码健壮性与上线风险类 | N6 代码生成 | 高危逻辑缺失、兼容性风险、上线方案缺失 |

**阶段与产出物映射**：表中"常见关联阶段"表示该问题类型最常见的表现层，但同一问题类型可能归因到不同阶段——例如 P4"代码规范与架构约束违反"可能归因到 N4（技术方案阶段缺少架构约束定义，属 D1 产出物质量问题）或 N6（架构约束已有但编码未遵守，属 D3 执行遵循问题）。实际归因以 Phase 2 决策树逆向定位结果为准，而非固定绑定到表中阶段。各分类的子类型、典型表现、根因及归因深推链详见附录 A（原第三章完整内容）。归因时若问题不匹配 P0~P5 已有分类，分析模型可自行定义新的问题类型（需给出明确定义、典型表现和改进方向）。

#### 3.3 维度二：过程传递（D2）

过程传递维度关注跨 Skill、跨 SubAgent、跨阶段的信息传递完整性。即便各阶段产出物质量达标，若信息在传递过程中丢失或失真，后续阶段仍将基于不完整信息生成代码。

##### 3.3.1 P6 上下文传递与信息断层类

**定义**：跨 Skill、跨 SubAgent、跨阶段的信息传递出现丢失或失真，导致后续阶段基于不完整信息生成代码。

**归因路径**：全链路 → work_status.md 完整性 → SubAgent 结果聚合质量

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P6-a | 跨 Skill 信息遗忘 | 前序阶段关键决策在后续 Skill 中未被引用 | Context 压缩导致关键信息被截断 |
| P6-b | SubAgent 结果断层 | 并行 SubAgent 产出未被正确聚合到主链 | SubAgent 结果聚合机制不完善 |
| P6-c | 阶段产出物不完整 | work_status.md / design.md 关键字段缺失 | 产出物格式校验不严格 |
| P6-d | 长程控制失效 | 多轮对话后偏离原始需求目标 | 目标锚定机制不足，缺乏全局一致性检查 |

**D2 与 D1 的关系**：P6 类问题的独特之处在于，各阶段产出物可能本身质量合格，但信息在阶段间传递时丢失。因此 D2 是独立于 D1 的正交维度。归因决策树中，当判定问题源于"信息传递失真"而非"产出物本身质量"时，归入 P6。

##### 3.3.2 各阶段问题分类

上游产出物本身质量 OK，但信息在跨阶段传递过程中丢失或失真。

| 阶段 | 一级问题分类 | 二级问题分类 | 说明 |
| --- | --- | --- | --- |
| **N5编码阶段** | **上下文传递与信息断层类** | **跨 Skill 信息遗忘** | 表现1：在 N5 场景下的典型表现是 N4 的 design.md 内容完整且正确，但由于 design.md 篇幅较长（复杂需求下可能几百行），在 N4→N5 的 Skill 切换过程中，LLM 的 Context 窗口发生压缩或截断，导致 design.md 后半部分的 D-xx 设计项、§5 接口定义、§6 存储变更、§7 稳定性要求等章节内容在 N5 的上下文中不可见。N5 并非有意遗漏这些设计项，而是根本"看不到"它们存在。<br>表现2：constraint-check.md 和 knowledge.md 作为 N4 的内部中间产物传递给 N5，但这两个文件的传递优先级低于 design.md，在 Context 空间紧张时更容易被丢弃，导致 MUST 约束和 SOP 条目在 N5 执行时完全缺席。判定方法是对比 N5 执行时实际可见的 Context 内容与 design.md 原文，如果原文中存在的 D-xx 在 Context 中找不到对应片段，跨Skill信息遗忘。 |
|  |  | **SubAgent 结果断层** | 在 N5 的执行流程中，阶段 1.5 门禁校验会启动一个独立的覆盖审查 SubAgent，该 SubAgent 审查 draft tasks.md 后返回结构化结果（PASS/PASS_WITH_RISK/BLOCKED + 未覆盖项列表）。P6-b 的问题是 SubAgent 正确识别了未覆盖的设计项或约束项并返回了 BLOCKED 或 PASS_WITH_RISK，但其返回结果未被主 Agent 正确接收或解析——比如 SubAgent 的 JSON 返回格式不完整、主链 Agent 未等待 SubAgent 完成就继续执行、或 SubAgent 返回的 `uncovered_design_items` 列表在聚合到主链时被截断。结果是审查做了、问题也发现了，但发现的问题没有传回来。 |
|  |  | **阶段产出物不完整** | design.md §6 定义了 DDL、Mapper、配置变更等存储层改动，但 tasks.md 中缺少对应的 B 类结构任务 |
|  |  | **长程控制失效** | 在 N5 场景下相对少见但也可能出现。如果在 N5 执行之前经历了多轮用户交互（比如 design.md 审批过程中用户提了多次修改意见），N5 启动时的上下文中可能混入了大量审批讨论记录，这些记录会稀释 design.md 的核心内容，导致 N5 在分析设计项时偏离了 design.md 最终审批版本的实际要求——比如将审批过程中被否决的方案误当作最终方案来拆分任务 |

#### 3.4 维度三：执行遵循（D3）

执行遵循维度是归因的前置筛选层。它回答的核心问题是：**AI 有没有按标准流程执行**？当某阶段已产出高质量的产出物（如详设 design.md 明确定义了接口契约），但后续阶段（如 N6 代码生成）未遵循该产出物进行编码，导致的采纳率损失应归入"执行遵循"维度，而非"产出物质量"维度。

##### 3.4.1 D3 与 D1/D2 的区分标准

| 判定条件 | 归入维度 | 举例 |
| --- | --- | --- |
| 产出物有缺陷，后续阶段基于有缺陷的输入行动 | D1 产出物质量 | design.md 未定义异常处理约束 → N6 生成的代码缺少异常处理 |
| 产出物质量 OK，但后续阶段未遵循该产出物 | D3 执行遵循 | design.md 已定义接口契约 → N6 生成代码时跳过该约束未引用 |
| 信息在传递过程中丢失（Context 截断等） | D2 过程传递 | design.md 已定义接口契约 → Context 压缩时被截断 → N6 未感知 |

##### 3.4.2 PCR 度量与归因联动

执行遵循维度通过 **PCR（Process Compliance Rate，流程遵循度）**指标进行量化度量，具体定义和计算方式详见第四章 4.1 节。PCR 是归因流程的前置筛选器：

8. PCR 不达标 → 优先定责为"执行遵循问题"（D3），改进方向为流程约束强化
9. PCR 达标 → 排除遵循因素，进入 D1+D2 归因（产出物质量 / 过程传递）
**典型场景**：N5 编码计划已明确要求"使用公司统一 RPC 框架"，但 N6 代码生成时未引用 N5 产出物中的该约束，自行选用了其他框架。此时 N5 产出物质量 OK（编码计划已写明），问题在于 N6 执行时未遵循 → 归入 D3。

#### 3.5 各阶段问题分类

归因问题分类原则：所有归因问题统一识别到是依赖输入质量问题还是执行过程问题，以从根据上输出后续改进建议，比如新增知识或者是优化执行流程、加强执行流程约束等。

每个阶段判定原则：判断是不是该层职责、具体是哪些环节的职责，属于则按该层问题类型归因，否则回溯到其他层。

归因方法论：引用**质量检验递进模型，**使用"缺陷定位漏斗“思路，**检验步骤为Q1（存在性）→ Q2（一致性）→ Q3（正确性）→ Q4（精确性）→ Q5（闭环性）**。

### 六、实现方案

#### 6.1 归因结果落库设计

归因分析（yx-agcr-attribution Skill）完成后，结果通过 HTTP POST 写入两张数据表，支持 AGCR 趋势查询、跨需求归因阶段统计、低采纳率需求筛选等场景。

##### 表设计总览

| 表名 | 唯一维度 | 职责 |
| --- | --- | --- |
| `yuanxi_agcr_attribution_run_result` | `(requirement_id, run_id, version)` | Run 级汇总：AGCR 指标、分布统计、报告链接 |
| `yuanxi_agcr_attribution_hunk_result` | `(run_result_id, hunk_id)` | Hunk 级明细：归因结论、证据链、分类信息 |

**多版本归因**：同一 `(requirement_id, run_id)` 可多次归因（代码持续演进、参数调整重跑），每次写入时 `version` 由服务端自动维护（MAX+1），调用方只传 `requirement_id + run_id`。Hunk 表通过 `run_result_id` 关联 Run 表主键。

```SQL
CREATE TABLE yuanxi_agcr_attribution_hunk_result (

  attribution_stage        VARCHAR(50)             COMMENT '最终归因阶段，如 N4 技术方案（根因定位结果，非时序首因）',

  created_at               DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
```
