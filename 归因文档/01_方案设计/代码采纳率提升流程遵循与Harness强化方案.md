# Agent代码采纳率提升：流程遵循与 Harness 强化方案

### 一、问题定位：为什么要从"事后归因"转向"事前遵循"

#### 1.1 现有方案的能力盲区

父方案建立了 P1-P8 八类问题分类、多层深推归因链、事前/事中/事后三时机以及评测自迭代机制，体系完整。但其重心落在"问题发生之后如何定位根因并人工优化 Skill"，存在三个结构性盲区。

**第一，预防能力偏弱**。采纳率归因分析方案的"事前预防（Pre-flight）"仅做 Spec 质量打分与就绪状态检查，本质上仍是"评分预警"，分数不达标时并不强制阻断流程，AI 仍可带病进入下一阶段，问题被推迟到事中事后才暴露。

**第二，约束不可执行**。采纳率归因分析方案的事中检测依赖"阶段完成自检"与"异常信号 Hook"，但这些 Hook 的产物是"归因诊断 + 改进建议"，是观测性的，不是强制性的。也就是说，系统看见了 AI 没遵守流程，却没有机制阻止它继续往下走。

**第三，迭代链路太长**。采纳率归因分析方案的优化路径是"归因集中度超阈值 → 人工触发 Skill 专项优化 → 下批需求验证"，从问题发生到 AI 行为改变要跨越多个需求周期，反馈太慢。真正高效的做法是把已知的正确行为前置固化为约束，让 AI 第一次就做对，而不是错了再回头改 Prompt。

#### 1.2 核心思路：把归因前置为约束

参考 Pipeline Workflow 的设计哲学——"**把 LLM 放进有约束的盒子里，让它在每个正确的位置做正确的事**"——本方案的核心动作是把父方案中"事后才发现的归因结论"前置转化为"事前就执行的强制约束"。归因体系回答的是"哪里容易错、为什么错"，这恰好为"应该在哪里设卡、卡什么"提供了精确依据。每一条高频归因链，都应当沉淀为一道可执行的事前 Guardrail 或 Checkpoint。

在现有采纳率归因分析方案基础上补齐三层能力：**流程遵循度的可度量、Harness 编排框架的强约束、AI 遵循能力的持续训练。最终让采纳率的提升主要来自"AI 一次做对"，而非"事后归因后人工修补"。**

#### 1.3 三层强化总览

**第一层是流程遵循度量化**，解决"AI 到底有没有按流程走"这个此前无法回答的问题，引入**流程遵循度（Process Compliance Rate）**作为与采纳率并列的核心指标。

**第二层是 Harness 强约束编排**，引入 Checkpoint 强制门控、Guardrail 规则兜底、阶段门禁（Stage Gate）三种**约束手段**，让不合规的产出物无法流入下一阶段。

**第三层是 AI 遵循能力训练**，把每次违规与归因结论沉淀为可复用的约束规则与上下文经验，通过 Workflow Memory 注入下一次执行，形成"违规一次、固化一次、下次不再犯"的正循环。

### 二、第一层：流程遵循度量化

#### 2.1 为什么要先度量遵循度

采纳率低有两种本质不同的成因：一种是 AI 认真按流程做了但能力不足（模型边界问题），另一种是 AI 根本没按流程做（遵循问题）。定义流程遵循度指标的目的是对流程遵循情况做出归因分析。

#### 2.1 流程遵循度（PCR）指标定义

流程遵循度衡量一次需求执行对标准流程的符合程度，按阶段加权计算。

| 维度 | 含义 | 度量方式 |
| --- | --- | --- |
| 阶段完整性 | N1-N8 与 QA 各必经节点是否实际执行 | 检查各节点产出物（current-state.md / requirement.md / design.md 等）是否存在且非空 |
| 产出物合规性 | 各阶段产出物是否满足格式与内容硬约束 | 对照各阶段 Schema 校验（AC 是否结构化、Spec-Design 映射是否完整等） |
| 门禁通过性 | 强制 Checkpoint 是否经过有效确认而非被绕过 | 检查 work_status.md 中的门禁状态字段与确认记录 |
| 约束遵守性 | 编码是否遵守注入的架构与规范约束 | 对照项目级/团队级约束清单做静态校验 |
| 顺序正确性 | 是否存在跳阶段、逆序执行 | 检查 work_status.md 阶段时间戳序列 |

流程遵循度 PCR = Σ(阶段权重 × 阶段合规得分)，其中需求澄清（N4）、技术方案（N6）两个强制 Checkpoint 阶段权重最高，因为父方案归因数据显示其下游放大效应最强。

#### 2.3 遵循度与采纳率的联合分析

遵循度指标与父方案采纳率指标联合使用，构成一张二维诊断矩阵，让治理动作不再一刀切。

|  | 遵循度高 | 遵循度低 |
| --- | --- | --- |
| 采纳率高 | 理想态：流程跑通且产出可用，沉淀为正样本经验 | 侥幸态：流程没走对但碰巧通过，需补强约束防止下次翻车 |
| 采纳率低 | 能力态：流程都走对了仍不达标，是真正的模型/方案能力问题，进入父方案 P1-P8 归因 | 遵循态：流程根本没走对，先治遵循（补 Checkpoint/Guardrail），不必急于归因 |

这张矩阵直接修正了父方案的一个隐含假设：并非所有低采纳率都需要归因。落在"遵循态"象限的需求，正确动作是强化事前约束而非事后深推；只有落在"能力态"象限的，才真正适用父方案的多层归因深推链。

### 三、第二层：Harness 强约束编排

本层借鉴 Pipeline Workflow 的 Harness 架构，把父方案 4.6 节中观测性的"异常信号 Hook"升级为可强制阻断的执行引擎。核心原则与参考方案一致：能用规则写死的绝不交给 LLM 自由发挥。

#### 3.1 三种约束手段

##### 3.1.1 Checkpoint 强制门控（人工决策门）

在错一步则下游全错的关键阶段设置不可跳过、不可自动审批的人工确认点。父方案已隐含此机制，本方案将其显式化并与遵循度强绑定。

强制确认点固定为两处：N4 需求澄清完成后（确认需求理解无歧义）、N6 技术方案完成后（确认方案可落地）。这两处对应父方案中 P1（需求理解偏差）与 P3（技术方案缺陷）两类高频源头根因，是采纳率下游放大效应的总闸门。AI 必须等待 RD 逐条确认或修正后才能继续；未经确认强行进入下一阶段的行为，直接判定为流程违规并记入 work_status.md。

产出物确认点（PR、测试用例等阶段）则可基于用户画像自动通过，不打扰人，与参考方案的分级门控策略一致。

##### 3.1.2 Guardrail 规则兜底（自动阻断门）

把父方案中"自检发现问题 → 输出归因建议"的观测性逻辑，升级为"校验不通过 → 自动阻断流程"的强制性逻辑。这是本方案与父方案最关键的差异：父方案看见问题但放行，本方案看见问题就拦截。

Guardrail 在每个阶段的产出物落盘时自动触发，按"系统默认 → 团队 → 用户 → 项目 → CLI 参数"五级配置逐层覆盖，每条规则支持 block / warn / skip 三种失败策略。

| 阶段 | Guardrail 规则示例 | 默认策略 | 对应归因防御 |
| --- | --- | --- | --- |
| N3 现状基线 | current-state.md 入口覆盖率 < 阈值 | warn | P2 现状基线偏差 |
| N4 需求澄清 | requirement.md 中存在无验收标准（AC）的需求项 | block | P1 需求理解偏差 |
| N6 技术方案 | design.md 中存在 Spec 项无对应设计映射 | block | P3 技术方案缺陷 |
| N8 编码实现 | 违反项目架构分层约束 / 单测覆盖率 < 80% | block | P4/P5 规范与健壮性 |
| 全链路 | 跨 Skill 关键上下文字段丢失 | block | P6 上下文断层 |
| QA3 | 接口用例未覆盖全部需求 AC | warn | P8 测试质量 |

block 策略下，产出物无法流入下一阶段，AI 必须修正后重新提交校验；warn 策略下放行但记录违规，纳入遵循度扣分与事后复盘。每一条 block 规则都直接来源于父方案的高频归因链——归因告诉我们错在哪，Guardrail 就在那里设卡。

##### 3.1.3 阶段门禁与应急预案（Stage Gate）

借鉴参考方案的 Stage Gate 应急预案引擎，让流程在卡住时既不失控也不卡死。当某阶段反复无法通过 Guardrail 时，自动触发分级处置而非简单报错退出。

编码连续失败 3 次自动回退到 N6 技术方案重新评审；测试连续失败 2 次携带失败上下文重试编码；同一阶段人工纠偏 ≥ 3 次升级人工接管。同时设 Budget 硬限制（全局最多 5 次重试、单阶段最多 2 次）防止无限循环。这些应急动作的触发条件与父方案 4.6 节的异常信号 Hook 完全复用，区别在于本方案让它们具备真正的流程控制权，而非仅输出诊断。

#### 3.2 从"观测 Hook"到"控制 Hook"的升级对照

| 能力项 | 父方案现状（观测性） | 本方案升级（控制性） |
| --- | --- | --- |
| 阶段自检 | 输出 quality_check 与归因建议 | 校验不通过则 block，阻断进入下一阶段 |
| 异常信号 | 写入 work_status.md 供事后分析 | 触发 Stage Gate 回退/重试/升级 |
| Spec 质量 | 打分预警，分低仍可继续 | 低于门禁分则 block，强制补全 |
| 约束注入 | 无强制注入 | 团队/项目约束自动注入并静态校验 |
| 反馈周期 | 跨需求周期人工优化 Skill | 本次执行内即时阻断与修正 |

#### 3.3 多级配置：让团队经验固化为流程约束

与参考方案一致，约束规则支持五级配置覆盖，使 TL 可在 Web 端配置团队级强制规则（如"支付相关需求必须经过安全扫描""核心交易链路单测覆盖率不低于 80%"），这些约束自动注入团队成员的执行流程，无需依赖每个人记得遵守。这把父方案中"靠 Prompt 提醒 AI 遵守"的软约束，变成了"由 Harness 强制执行"的硬约束。

### 四、第三层：AI 遵循能力持续训练

强约束解决"这一次做对"，持续训练解决"以后都做对"，避免 AI 反复在同一处被 Guardrail 拦截。

#### 4.1 违规驱动的约束沉淀

每一次 Guardrail block 或 Checkpoint 人工修正，都是一条高质量的负样本。本方案要求将其结构化沉淀：记录违规阶段、违规类型（对应 P1-P8）、被拦截的产出物快照、人工修正后的正确产出物。这批数据有两个去向：一是反哺父方案的归因模型校正库，二是提炼为新的 Guardrail 规则或 Prompt 约束，前置到下一次执行。

#### 4.2 Workflow Memory 注入正确行为

借鉴参考方案的 Workflow Memory 机制，让正确的遵循行为可被检索复用。每个 Skill 完成任务后沉淀经验（含标签、置信度、关联场景），进入类似场景时自动注入上下文，高置信度经验逐步升级为强制规则，低置信度经验逐步衰减。同时维护每位工程师的 Profile（架构偏好、review 严格度），使约束既统一又能贴合个人与团队习惯。如此，团队踩过的坑会"长在流水线上"，新成员一上手即享受团队积累的遵循经验。

#### 4.3 闭环：让遵循能力越跑越强

本方案与父方案共同构成一个双层自迭代闭环。下层是父方案的归因自迭代（AI 归因 → 人工校正 → 模型更新）；上层是本方案的遵循自迭代（违规拦截 → 约束沉淀 → 经验注入 → 下次一次做对）。上层把下层的归因结论持续转化为事前约束，使需要归因的需求比例（落在"能力态"象限）逐步上升、需要治理遵循的需求比例（落在"遵循态"象限）逐步下降——这正是方案成功的标志：当大部分采纳率问题都变成纯能力问题时，说明流程遵循已不再是瓶颈。

### 五、与父方案的衔接与实施路径

#### 5.1 章节级衔接关系

| 父方案章节 | 本方案对应强化 |
| --- | --- |
| 1.2 分析时机（事前/事中/事后） | 把"事前预防"从评分升级为强制门控（三、Checkpoint/Guardrail） |
| 3.2 问题分类（P1-P8） | 每类高频问题对应一道事前 Guardrail（3.1.2 表） |
| 4.6 事中归因触发点 | 观测 Hook 升级为控制 Hook（3.2 对照表） |
| 5 归因指标体系 | 新增流程遵循度 PCR，与采纳率联合诊断（二） |
| 6.3 自迭代触发条件 | 新增遵循自迭代上层闭环（4.3） |

#### 5.2 分阶段落地节奏

第一阶段（先可观测）：上线流程遵循度 PCR 指标与二维诊断矩阵，先把遵循问题与能力问题拆开，所有 Guardrail 先以 warn 策略运行，只记录不阻断，用于校准阈值与统计违规分布。

第二阶段（再强约束）：将 N4、N6 两处 Checkpoint 与高频违规对应的 Guardrail 切换为 block 策略，正式启用 Stage Gate 应急预案，让不合规产出物无法流入下游。阈值取自第一阶段统计与父方案历史归因数据。

第三阶段（后自进化）：打通违规沉淀与 Workflow Memory 注入闭环，将稳定的团队约束固化为多级配置，逐步把人工 Checkpoint 中可自动判定的部分下沉为 Guardrail，使遵循能力进入自我强化轨道。

#### 5.3 度量成功的标准

方案成效以三个指标联合判断：流程遵循度 PCR 持续上升并稳定在高位；采纳率提升中"由遵循改善贡献的部分"可被归因区分出来且占比领先；落在"能力态"象限（遵循高但采纳率低）的需求占比相对上升，表明剩余问题已收敛为纯模型能力问题，可干净地交给父方案归因体系处理。
