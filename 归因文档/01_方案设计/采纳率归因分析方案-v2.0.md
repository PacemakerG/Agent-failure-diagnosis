## Agent代码采纳率归因分析方案 v2.0

> 版本：v2.0 | 更新日期：2025-06
> 基于：Agent AI-SDLC 研发全流程（10节点主链 + SubAgent + PM/QA并行）
> 核心升级：引入**多层归因深推链**,支持问题表象→中间根因→源头根因的链式追溯

---

### 一、背景与目标

#### 1.1 现状问题

Agent当前代码采纳率（AGCR）约 75%，核心瓶颈已通过 Spec v2.0 迭代思路定位为"Spec 不够细"。但在实际工程实践中，**同一个低采纳率问题往往有多层成因**：

- 表象层：AI 生成代码质量差（P8 QA 测试未覆盖）
- 中间层：详设 Spec 不够细（P3 详设缺陷）
- 源头层：需求 Spec 本身存在歧义（P1 需求质量不足）

单层归因只能定位"最近的原因"，无法指导根本性改进。v2.0 方案引入**归因深推链（Attribution Chain）**，支持从问题表象逐层追溯到源头根因，并在每一层给出对应的改进动作。

#### 1.2 目标

1. 建立覆盖Agent全流程（n1-n10 + SubAgent + PM/QA并行）的归因分类体系
2. 支持多层归因深推，识别跨阶段的因果依赖链
3. 提供事前预防、事中检测、事后分析三位一体的归因闭环
4. 输出可量化的归因指标，驱动 Spec v2.0 三阶段迭代落地

### 三、归因分类体系 v2.0

#### 3.1 分类总览

| 分类 | 名称 | 所属阶段 | 典型表现 |
| --- | --- | --- | --- |
| P0 | 上下文理解缺失 | n1 现状理解 | 代码库理解偏差，生成代码与现有架构不符 |
| P1 | 需求 Spec 质量不足 | n2 需求 Spec | 需求描述模糊、验收标准缺失、边界条件未定义 |
| P2 | 需求理解偏差 | n2→n3 | AI 对需求的理解与 PM 意图不一致 |
| P3 | 详设 Spec 质量不足 | n3 详设 Spec | 接口定义不完整、数据结构缺失、逻辑分支未覆盖 |
| P4 | 代码生成质量问题 | n4 代码生成 | 语法错误、逻辑错误、风格不符 |
| P5 | 自检覆盖不足 | n5 代码自检 | 自检未发现已知问题 |
| P6 | 评审有效性不足 | n6 代码评审 | 评审意见未被正确理解或执行 |
| P7 | 单测覆盖率不足 | n7 单测生成 | 关键路径未覆盖、边界用例缺失 |
| P8 | 集成测试问题 | n8 集成测试 | 测试用例设计不足、环境问题、QA 并行链路缺失 |
| P9 | 修复引入新问题 | n9 问题修复 | 修复代码引入回归问题 |
| P10 | 合并冲突 | n10 代码合并 | 合并时产生冲突导致代码被覆盖或丢失 |

#### 3.2 各分类详细说明

##### P0：上下文理解缺失

**触发条件**：生成代码与现有代码库架构、命名规范、依赖关系不符

**子分类**：

1. P0-a：代码库结构理解偏差（SA-Context 未正确解析模块依赖）
2. P0-b：技术栈版本理解错误（如使用了已废弃的 API）
3. P0-c：业务上下文缺失（不了解现有业务逻辑约束）

**归因深推链**：

```Plain Text
P0（上下文缺失）
  └─► 根因A：n1 现状理解阶段 SA-Context 输入不足（代码库索引不完整）
  └─► 根因B：P1 需求 Spec 未包含技术约束说明
```

---

##### P1：需求 Spec 质量不足

**触发条件**：AI 生成代码功能正确但不满足业务需求，或需求理解存在歧义

**子分类（对齐 Spec v2.0 迭代思路）**：

4. P1-a：需求描述过于模糊（缺少具体场景描述）
5. P1-b：验收标准缺失（无法判断生成代码是否满足需求）
6. P1-c：边界条件未定义（异常场景、极端值未说明）
7. P1-d：现状理解不足导致 Spec 偏差（**新增**，对应 Spec v2.0 第一步：现状理解）
8. P1-e：需求 Spec 模板结构不完整（**新增**，对应 Spec v2.0 第二步：需求 Spec 模板化）
9. P1-f：PM 与 RD 对需求理解不一致（**新增**，对应 pm1/pm2 并行链路缺失）

**归因深推链**：

```Plain Text
P1（需求 Spec 质量不足）
  └─► 根因A：n1 现状理解阶段输出质量不足（→ P0）
  └─► 根因B：PM 并行链路（pm1）未完成需求澄清
  └─► 根因C：Spec 模板未标准化，关键字段缺失
```

---

##### P2：需求理解偏差

**触发条件**：Spec 文档质量尚可，但 AI 对需求的解读与预期不符

**子分类**：

10. P2-a：自然语言歧义（同一描述有多种合理解读）
11. P2-b：隐含假设未显式化（PM/RD 认为"显而易见"但 AI 不知道）
12. P2-c：跨文档引用理解错误（Spec 引用了其他文档但 AI 未读取）
13. P2-d：SA-Spec SubAgent 理解偏差（**新增**，SA-Spec 在 Spec 细化时引入偏差）

**归因深推链**：

```Plain Text
P2（需求理解偏差）
  └─► 根因A：P1-a/b/c Spec 本身质量问题（需求不清导致理解偏差）
  └─► 根因B：SA-Spec 上下文窗口不足，未读取完整 Spec
  └─► 根因C：pm2 验收标准确认环节缺失
```

---

##### P3：详设 Spec 质量不足

**触发条件**：需求理解正确，但详细设计文档不足以指导代码生成

**子分类（对齐 Spec v2.0 第三步：详设 Spec）**：

14. P3-a：接口定义不完整（参数类型、返回值、错误码缺失）
15. P3-b：数据结构定义缺失（数据库表结构、DTO 定义未说明）
16. P3-c：逻辑分支未覆盖（if/else 分支、状态机转换未完整描述）
17. P3-d：依赖关系未说明（调用链、外部服务依赖未标注）
18. P3-e：详设 Spec 与需求 Spec 不一致（**新增**，对应 Spec v2.0 一致性校验需求）

**归因深推链**：

```Plain Text
P3（详设 Spec 质量不足）
  └─► 根因A：P1 需求 Spec 不完整，详设无法细化
  └─► 根因B：P2 需求理解偏差，详设方向错误
  └─► 根因C：SA-Spec 在详设阶段未调用足够的上下文工具
  └─► 根因D：n1 现状理解不足，详设与现有代码架构冲突
```

---

##### P4：代码生成质量问题

**触发条件**：Spec 完整，但生成代码本身存在质量问题

**子分类**：

19. P4-a：语法/编译错误
20. P4-b：逻辑错误（算法实现有误）
21. P4-c：代码风格不符（命名规范、格式规范）
22. P4-d：性能问题（不必要的循环、N+1 查询等）
23. P4-e：安全漏洞（SQL 注入、XSS 等）

**归因深推链**：

```Plain Text
P4（代码生成质量）
  └─► 根因A：P3 详设 Spec 不足，SA-CodeGen 缺少足够约束
  └─► 根因B：P0 上下文缺失，生成代码与现有代码风格不符
  └─► 根因C：模型能力边界（当前模型对特定语言/框架支持不足）
```

---

##### P5：自检覆盖不足

**触发条件**：n5 自检阶段未发现已知问题，问题流转到后续阶段

**子分类**：

24. P5-a：自检规则集不完整（未覆盖特定类型的错误）
25. P5-b：自检深度不足（只做语法检查，未做逻辑检查）
26. P5-c：自检与 Spec 脱节（未对照 Spec 验证功能完整性）

**归因深推链**：

```Plain Text
P5（自检覆盖不足）
  └─► 根因A：P3 详设 Spec 缺少可验证的验收标准
  └─► 根因B：自检 Prompt 未包含 Spec 对照检查指令
```

---

##### P6：评审有效性不足

**触发条件**：代码评审阶段未有效拦截问题

**子分类**：

27. P6-a：评审意见过于笼统（AI 无法理解如何修改）
28. P6-b：评审未覆盖关键逻辑
29. P6-c：评审意见执行偏差（AI 修改方向与评审意图不符）

**归因深推链**：

```Plain Text
P6（评审有效性不足）
  └─► 根因A：P3 详设 Spec 缺失，评审缺少参照标准
  └─► 根因B：SA-Review 上下文不足，未读取完整 Spec 进行对照评审
```

---

##### P7：单测覆盖率不足

**触发条件**：单测未覆盖关键路径，导致问题在集成测试或生产环境暴露

**子分类**：

30. P7-a：核心业务路径未覆盖
31. P7-b：边界条件/异常路径未覆盖
32. P7-c：Mock 设计不合理（过度 Mock 导致测试失真）

**归因深推链**：

```Plain Text
P7（单测覆盖率不足）
  └─► 根因A：P3-c 逻辑分支未在详设中定义，单测无法覆盖
  └─► 根因B：P1-c 边界条件未在需求 Spec 中定义
  └─► 根因C：QA 并行链路（qa1）测试用例设计未同步给单测生成阶段
```

---

##### P8：集成测试问题

**触发条件**：集成测试阶段发现问题，导致代码被拒绝或大幅修改

**子分类（新增 QA 并行链路归因）**：

33. P8-a：测试用例设计不足（qa1 阶段用例覆盖不全）
34. P8-b：测试环境问题（环境配置差异导致测试失败）
35. P8-c：QA 并行链路缺失（qa1-qa4 未与 RD 主链同步）
36. P8-d：回归测试覆盖不足（qa3 未覆盖历史功能）
37. P8-e：验收标准不一致（qa4 验收标准与 pm2 确认的标准不一致）

**归因深推链（多层深推示例）**：

```Plain Text
P8（集成测试问题）
  ├─► 中间层A：P7 单测覆盖不足（问题未在单测阶段拦截）
  │     └─► 源头层：P3-c 逻辑分支未在详设中定义
  │           └─► 更深层：P1-c 边界条件未在需求 Spec 中定义
  ├─► 中间层B：P5 自检未发现问题
  │     └─► 源头层：P3 详设 Spec 缺少可验证标准
  ├─► 中间层C：P8-c QA 并行链路缺失
  │     └─► 源头层：流程设计问题（qa1 测试用例未在 n3 阶段同步）
  └─► 直接根因：P8-a 测试用例设计不足（qa1 阶段问题）
```

---

##### P9：修复引入新问题

**触发条件**：n9 修复阶段的代码引入了新的回归问题

**子分类**：

38. P9-a：修复范围过大（修改了不相关的代码）
39. P9-b：修复方向错误（对问题根因理解有误）
40. P9-c：修复后未重新自检/单测

**归因深推链**：

```Plain Text
P9（修复引入新问题）
  └─► 根因A：P8 问题定位不准确，修复方向错误
  └─► 根因B：P3 详设 Spec 不完整，修复时缺少约束
  └─► 根因C：P5/P7 自检/单测未在修复后重新执行
```

---

##### P10：合并冲突

**触发条件**：n10 代码合并阶段产生冲突

**子分类**：

41. P10-a：并行开发冲突（多个 AI Agent 并行修改同一文件）
42. P10-b：合并策略错误（冲突解决时选择了错误的版本）
43. P10-c：依赖版本冲突

**归因深推链**：

```Plain Text
P10（合并冲突）
  └─► 根因A：P3 详设 Spec 未明确模块边界，导致并行开发冲突
  └─► 根因B：SubAgent 并行执行时缺少协调机制
```

### 四、多层归因深推机制

#### 4.1 归因深推链设计原则

**核心思想**：每个问题分类（Px）都可能是另一个问题的"中间层"，而非"源头层"。归因分析需要沿着因果链向上游追溯，直到找到**可干预的源头根因**。

**深推终止条件**：

1. 到达 n1（现状理解）或 n2（需求 Spec）阶段 → 流程最上游
2. 到达"模型能力边界"→ 需要模型升级，非流程可改进
3. 到达"外部依赖"→ 如第三方 API 变更，非内部可控

#### 4.2 归因深推链全景图

```Plain Text
源头层（n1-n2）          中间层（n3-n7）          表象层（n8-n10）
─────────────────────────────────────────────────────────────────
P0 上下文缺失 ──────────────────────────────────► P4 代码质量
                                                  ↑
P1 需求Spec质量 ──────► P3 详设Spec质量 ─────────► P4 代码质量
      │                      │                    │
      │                      ├──────────────────► P5 自检不足
      │                      │                    │
      │                      ├──────────────────► P7 单测不足
      │                      │                    │
      ▼                      ▼                    ▼
P2 需求理解偏差 ──────► P6 评审不足 ────────────► P8 集成测试
                                                  │
                        P9 修复引入问题 ◄──────────┘
                              │
                              ▼
                        P10 合并冲突
```

#### 4.3 高频深推链（Top 5）

根据Agent当前 75% 采纳率的问题分布，以下是最常见的多层归因链：

**链路 1：Spec 质量链（最高频）**

```Plain Text
P8/P9（测试/修复问题）→ P3（详设不足）→ P1（需求Spec不足）→ P1-d（现状理解不足）
```

改进动作：Spec v2.0 三阶段迭代（现状理解→需求Spec→详设Spec）

**链路 2：上下文缺失链**

```Plain Text
P4（代码质量差）→ P0（上下文缺失）→ n1（现状理解阶段 SA-Context 输入不足）
```

改进动作：增强 SA-Context 代码库索引能力，n1 阶段强制输出现状理解报告

**链路 3：QA 并行缺失链**

```Plain Text
P8-c（QA并行缺失）→ P7（单测不足）→ P3-c（逻辑分支未定义）→ P1-c（边界条件未定义）
```

改进动作：qa1 测试用例设计与 n3 详设 Spec 同步，建立 QA-RD 并行协作机制

**链路 4：评审失效链**

```Plain Text
P6（评审无效）→ P3（详设缺少验收标准）→ P1-b（验收标准缺失）
```

改进动作：P1/P3 Spec 模板中强制包含可验证的验收标准

**链路 5：SubAgent 理解偏差链**

```Plain Text
P4（代码质量）→ P2-d（SA-Spec理解偏差）→ P1-e（Spec模板不完整）
```

改进动作：SA-Spec 增加 Spec 完整性自检步骤，输出结构化 Spec JSON

### 五、归因分析三位一体闭环

#### 5.1 事前预防（Pre-flight）

在代码生成前，通过 Spec 质量检查预防归因问题：

| 检查项 | 对应归因 | 检查方式 |
| --- | --- | --- |
| 现状理解报告完整性 | P0/P1-d | SA-Context 输出结构化报告，包含模块依赖图 |
| 需求 Spec 完整性评分 | P1 | 基于 Spec v2.0 模板，检查必填字段覆盖率 |
| 详设 Spec 可验证性 | P3 | 检查接口定义、数据结构、逻辑分支覆盖率 |
| QA 并行链路就绪 | P8-c | qa1 测试用例是否在 n3 完成前同步 |

**Spec 质量评分模型**：

```Plain Text
Spec质量分 =
  现状理解完整度(20%) +
  需求描述清晰度(25%) +
  验收标准可验证性(25%) +
  详设接口完整度(20%) +
  边界条件覆盖率(10%)
```

#### 5.2 事中检测（In-flight）

在研发流程各节点实时触发归因检测：

| 触发节点 | 检测内容 | 归因输出 |
| --- | --- | --- |
| n4 代码生成后 | 代码与 Spec 一致性检查 | P3/P4 |
| n5 自检后 | 自检覆盖率统计 | P5 |
| n6 评审后 | 评审意见执行率 | P6 |
| n7 单测后 | 单测覆盖率报告 | P7 |
| n8 集成测试后 | 测试失败根因分析 | P8 + 深推链 |

**事中归因触发规则**：

4. 自检发现问题 → 自动触发 P3/P4 归因分析
5. 单测覆盖率 < 80% → 自动触发 P3-c/P1-c 归因深推
6. 集成测试失败 → 自动触发 P8 多层深推链分析

#### 5.3 事后分析（Post-flight）

代码合并后，基于全流程数据进行归因统计：

**归因数据采集点**：

7. 每个节点的输入/输出质量评分
8. 问题首次发现节点 vs 问题根因节点
9. 修复成本（修复轮次 × 修改行数）

**归因报告维度**：

1. 按归因分类统计（P0-P10 分布）
2. 按深推链统计（源头根因分布）
3. 按研发阶段统计（问题在哪个阶段被发现 vs 在哪个阶段产生）
4. 按 Spec 质量分统计（Spec 质量与采纳率的相关性）
