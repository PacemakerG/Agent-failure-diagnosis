### 一、方案概述

#### 1.1 背景与目标

Agent（Agent）是公司 AI-SDLC 研发流程的核心 Agent 系统，通过多阶段 Skill 驱动完成从需求入口到代码交付的全链路自动化研发。**代码采纳率**（AGCR，AI-Generated Code Ratio）是衡量Agent产出质量的核心指标，反映 AI 生成代码被 RD 实际采纳的比例。

本方案旨在建立一套**系统化的采纳率归因分析框架**，实现：

1. **辅助分析**：帮助 RD/PM/QA 快速定位采纳率低的根因，指导改进方向
2. **工具评测**：通过归因结果对Agent各 Skill 进行量化评测，支持 Skill 自迭代
3. **可扩展性**：框架设计支持新增归因维度和问题类型，随流程演进持续完善

#### 1.2 分析时机

**事中分析**：Skill 执行过程中内嵌初稿归因框架，实时输出问题分类 + 归因路径 + 改进建议。各阶段的事中归因触发点详见第四章 4.4 节。

**事后分析**：需求交付后，基于完整 context（需求记录、AI-trace、代码 diff）进行深度归因复盘。

#### 1.3 核心分析逻辑

采纳率归因遵循以下分析路径：

```Plain Text
采纳率低 → 识别被修改/拒绝的代码片段 → 定位所属研发阶段 → 匹配问题分类 → 追溯根因 → 输出改进建议
```

**分析结论要求**：每次归因分析必须给出明确的分析结论，包括：主要问题类型占比、最高优先级根因、针对性改进建议，不能仅列举问题而不给出判断。

### 三、问题分类体系（严格正交）

#### 3.1 分类设计原则

问题分类遵循 **MECE 原则**（Mutually Exclusive, Collectively Exhaustive）：

4. **互斥性**：每个被修改/拒绝的代码片段只归属一个一级类型
5. **完备性**：所有采纳率损失场景均可被覆盖
6. **可操作性**：每个分类对应明确的改进方向和责任 Skill
7. **通用归因类**：设立独立的通用归因类（P7），承接跨阶段、跨 Skill 的系统性问题，避免强行归入单一类别导致归因失真

#### 3.2 一级问题分类（8 类，严格正交）

##### 【P1】需求理解偏差类

**定义**：AI 对需求的理解与 RD/PM 预期存在偏差，导致实现方向错误或范围不符。

**重要说明**：P1 类明确区分两种根本不同的根因——"需求 Spec 本身质量不足"（P1-d/P1-e/P1-f）与"AI 对高质量 Spec 的理解偏差"（P1-a/P1-b/P1-c）。这两类问题的改进方向完全不同：前者应优化 Spec 生成流程，后者才应优化模型理解能力。

**归因路径**：需求澄清阶段（N4）→ requirement.md 质量 → PM 并行澄清充分性

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P1-a | 功能边界偏差 | 多实现或少实现功能点 | In/Out Scope 未显式表达，需求边界模糊 |
| P1-b | 业务语义误解 | 实现逻辑与业务规则不符 | 需求描述细节模糊，AI 按自身理解填充 |
| P1-c | 优先级/约束遗漏 | 忽略非功能性需求（性能/安全/兼容） | 需求 Spec 中隐含约束未被显式提取 |
| P1-d | 需求 Spec 质量不足 | 需求 Spec 本身存在歧义、冲突或覆盖不全 | 需求 Spec 生成流程质量不足，非 AI 理解问题 |
| P1-e | Spec 格式/信息密度不足 | html 化 Spec 信息密度低，AI 理解准确性下降 | Spec 格式质量不足，未达到 html 化标准 |
| P1-f | 多角度澄清缺失 | PM/RD/QA 某一视角的澄清缺失，导致实现偏差 | 需求澄清未覆盖 PM（业务契约）/RD（技术可行性）/QA（可测试性）三方视角 |

**归因决策树**：

1. 若 Spec 本身有歧义/冲突 → P1-d（优化 Spec 生成流程）
2. 若 Spec 格式/信息密度不足 → P1-e（优化 html 化质量）
3. 若某一视角澄清缺失 → P1-f（补充对应视角澄清）
4. 若 Spec 质量良好但 AI 理解偏差 → P1-a/P1-b/P1-c（优化模型理解能力）

**分析结论示例**：若 P1-d 占比 > 20%，核心结论为"需求 Spec 生成质量不足"，建议优化 yx-pm-analyze-requirement 的 Spec 质量自检机制，而非调整模型参数。

##### 【P2】现状基线偏差类

**定义**：AI 对代码现状的理解存在偏差，导致基于错误现状生成方案或代码。

**归因路径**：现状基线扫描阶段（N3）→ current-state.md 准确性 → evidence.md 完整性

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P2-a | 入口路径错误 | 找错业务入口，基于错误调用链生成代码 | 代码扫描覆盖不全，入口识别算法偏差 |
| P2-b | 影响范围遗漏 | 改了 A 未评估对 B 的影响，联调才发现 | 调用链分析深度不足，横向影响评估缺失 |
| P2-c | 冲突&兼容性识别失败 | 未识别新旧逻辑冲突或接口兼容性问题 | 冲突识别能力不足，兼容性分析缺失 |
| P2-d | 业务流后链路影响遗漏 | 改动影响了业务流下游链路，但未被识别 | 业务流后链路分析缺失，"谁是当前流程的后链路"未被分析 |

**分析结论示例**：若 P2-d 频繁出现，核心结论为"业务流后链路分析能力薄弱"，建议在 yx-current-state-baseline 中增加业务流后链路追踪能力。

##### 【P3】技术方案设计缺陷类

**定义**：技术方案设计阶段产出的 design.md 存在设计缺陷，导致编码阶段产出不可用。

**归因路径**：技术方案设计阶段（N6）→ design.md 质量 → SubAgent 设计质检充分性

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P3-a | Spec-详设断层 | 需求 Spec 有但详设未体现，功能点遗漏 | 需求到设计的映射追踪机制不完善 |
| P3-b | 接口契约缺失 | 跨仓/跨包接口定义不清，联调失败 | design-interface.md 覆盖不全 |
| P3-c | 技术路径误判 | 选择了不适合当前架构的技术方案 | 知识加载不充分，架构约束未被感知 |
| P3-d | 设计质检漏检 | SubAgent 质检通过但方案仍有缺陷 | 质检维度覆盖不全（业务正确性/技术完整性/可实现性）或质检标准过松 |
| P3-e | 详设推导链不完整 | 详设缺少从问题拆解到实现约束的完整推导逻辑 | yx-plan 推导链断裂：问题拆解→架构决策→核心链路→实现约束四步中存在跳跃 |

**P3-d 与 P3-e 的区分**：P3-d 是"质检通过但方案有缺陷"（质检标准问题），P3-e 是"推导逻辑本身不完整"（推导过程问题）。

**分析结论示例**：若 P3-e 高发，核心结论为"详设推导链质量不足"，建议在 yx-plan 中增加推导链完整性自检（需求覆盖度、高危场景充分性、约束覆盖度）。

##### 【P4】代码规范与架构约束违反类

**定义**：生成代码功能正确，但不符合团队架构约束、编码规范或中间件使用规范。

**归因路径**：编码实现阶段（N8）→ 知识加载质量（N5）→ 规范知识库完整性

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P4-a | 架构分层违反 | 跨层调用、分包不规范、基建未复用 | 架构约束知识未被加载或加载不完整 |
| P4-b | 中间件使用不当 | 缓存/MQ/RPC 使用方式不符合规范 | 中间件规范文档未被 yx-middleware-learn 覆盖 |
| P4-c | 编码风格不一致 | 命名/注释/异常处理风格与存量代码不符 | yx-code-learn 对编码风格的提取能力不足 |
| P4-d | 流程复用缺失 | 重复实现已有公共流程/工具类 | 知识加载阶段未识别可复用组件 |

**分析结论示例**：若 P4 类问题集中在 P4-a/P4-b，核心结论为"领域知识加载质量不足"，建议优化 yx-domain-knowledge 和 yx-middleware-learn 的知识提取深度。

##### 【P5】代码健壮性与上线风险类

**定义**：生成代码存在高危逻辑缺陷或缺乏上线保障措施，无法直接上线。

**归因路径**：编码实现阶段（N8）→ TDD 覆盖充分性 → 自优化（yx-self-optimization）质量

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P5-a | 高危逻辑缺失 | 事务边界/异常处理/并发安全未覆盖 | A 类任务 TDD 用例设计不充分 |
| P5-b | 兼容性风险 | 历史数据兼容/接口向后兼容未处理 | 现状基线中历史数据分析不足 |
| P5-c | 上线方案缺失 | 无灰度开关/回滚方案/监控告警 | 上线方案设计未纳入编码计划 |
| P5-d | 边界条件遗漏 | 空值/越界/超时等边界场景未处理 | 单测覆盖矩阵设计不完整 |

**分析结论示例**：若 P5-c 高发，核心结论为"上线工程化能力薄弱"，建议在 yx-writing-plans 中强制要求上线方案任务项。

##### 【P6】上下文传递与信息断层类

**定义**：跨 Skill、跨 SubAgent、跨阶段的信息传递出现丢失或失真，导致后续阶段基于不完整信息生成代码。

**归因路径**：全链路 → work_status.md 完整性 → SubAgent 结果聚合质量

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P6-a | 跨 Skill 信息遗忘 | 前序阶段关键决策在后续 Skill 中未被引用 | Context 压缩导致关键信息被截断 |
| P6-b | SubAgent 结果断层 | 并行 SubAgent 产出未被正确聚合到主链 | SubAgent 结果聚合机制不完善 |
| P6-c | 阶段产出物不完整 | work_status.md / design.md 关键字段缺失 | 产出物格式校验不严格 |
| P6-d | 长程控制失效 | 多轮对话后偏离原始需求目标 | 目标锚定机制不足，缺乏全局一致性检查 |

**分析结论示例**：若 P6-a 与 P6-d 同时出现，核心结论为"长程上下文管理能力不足"，建议引入全局目标追踪机制。

##### 【P7】通用归因类（系统性/外部因素）

**定义**：无法归入上述单一阶段的系统性问题，或由外部因素（模型能力、基础设施）导致的采纳率损失。

**设立意义**：通用归因类是框架的"兜底层"，承接跨阶段的系统性问题，避免强行归入单一类别导致归因失真。仅在无法归入 P1-P6/P8 时使用。

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P7-a | 模型能力边界 | 特定技术领域（如复杂 SQL/算法）生成质量普遍低 | 基础模型在该领域的能力上限 |
| P7-b | 需求本身不可实现 | 需求存在技术矛盾或资源约束，无法自动化实现 | 需求可行性评估缺失 |
| P7-c | 人工偏好差异 | 代码功能正确但 RD 个人风格偏好不同 | 非质量问题，属于风格对齐问题，建议从采纳率计算中剔除 |
| P7-d | 环境/基础设施问题 | 因测试环境、依赖服务不稳定导致代码被修改 | 外部依赖问题，非 AI 生成质量问题 |

##### 【P8】测试质量驱动的采纳率损失类

**定义**：QA 并行链路（QA1-QA4）产出的测试用例质量不足，导致 N9 阶段 AICC 覆盖率不达标，进而引发代码被修改或补充。

**设立意义**：P8 类专门覆盖"因测试质量不足导致的采纳率损失"，与 P5（代码健壮性风险）的区别在于：P5 是代码本身的质量问题，P8 是测试用例设计不足导致漏测后被发现的问题。

**归因路径**：QA 并行链路（QA1-QA4）→ ec-test-cases.md / test-cases.md 质量 → N9 AICC 覆盖率

| 编号 | 子类型 | 典型表现 | 根因 |
| --- | --- | --- | --- |
| P8-a | EC 黑盒用例覆盖不全 | 关键业务场景未被 EC 用例覆盖，漏测后代码被修改 | QA1/QA2 阶段 EC 用例设计不充分，场景覆盖不全 |
| P8-b | 接口用例设计缺陷 | 接口用例未覆盖边界场景，AICC 不达标 | QA3 阶段接口用例设计质量不足 |
| P8-c | 测试用例与需求脱节 | 测试用例未覆盖需求 AC，导致验收失败后代码被修改 | QA 并行链路与需求澄清阶段（N4）的同步机制不完善 |
| P8-d | 测试覆盖率指标虚高 | AICC 覆盖率达标但实际业务场景未覆盖 | 测试用例设计偏向覆盖率指标而非业务场景完整性 |

**分析结论示例**：若 P8-a 高发，核心结论为"EC 黑盒用例覆盖不足"，建议优化 QA1/QA2 阶段的用例生成策略，增加业务场景覆盖度检查。

#### 3.3 问题分类正交性验证

| 分类 | 核心区分维度 | 与相邻类的边界 |
| --- | --- | --- |
| P1 需求理解偏差 | 需求澄清阶段，理解层面（含 Spec 质量） | 区别于 P2：P1 是"理解需求错了"，P2 是"理解现状错了"；P1-d/e/f 是 Spec 质量问题，P1-a/b/c 是 AI 理解问题 |
| P2 现状基线偏差 | 现状扫描阶段，认知层面 | 区别于 P3：P2 是"输入信息错了"，P3 是"设计逻辑错了"；P2-d 专指业务流后链路遗漏 |
| P3 技术方案缺陷 | 设计阶段，方案层面 | 区别于 P4：P3 是"方案本身有问题"，P4 是"方案正确但代码不规范"；P3-e 专指推导链断裂 |
| P4 规范约束违反 | 编码阶段，规范层面 | 区别于 P5：P4 是"风格/规范问题"，P5 是"健壮性/安全问题" |
| P5 健壮性风险 | 编码阶段，质量层面 | 区别于 P8：P5 是代码本身的质量缺陷，P8 是测试用例不足导致漏测后被发现 |
| P6 上下文断层 | 全链路，信息传递层面 | 区别于其他：P6 是"信息传递问题"，其他是"生成质量问题" |
| P7 通用归因 | 系统性/外部因素 | 兜底类，仅在无法归入 P1-P6/P8 时使用 |
| P8 测试质量损失 | QA 并行链路，测试覆盖层面 | 区别于 P5：P8 根因在 QA 链路，P5 根因在编码链路 |

### 四、归因分析框架

#### 4.1 分析流程

```Plain Text
Step 1: 数据采集
  ├── 获取代码 diff（被修改/拒绝的代码片段）
  ├── 关联 yuanxi_observability_sessions（会话 context）
  ├── 提取 AI-trace（模型调用链路）
  └── 读取 work_status.md（产出物路径，用于代码片段-Skill 关联）

Step 2: 片段定位
  ├── 识别每个被修改片段所属的研发阶段
  ├── 通过 work_status.md 产出物路径关联对应 Skill 和执行节点
  └── 标记修改类型（功能修改/风格修改/删除/重写）

Step 3: 问题分类
  ├── 按 P1-P8 分类体系进行一级分类
  ├── 进一步细化到二级子类型
  └── 标记置信度（高/中/低）

Step 4: 根因追溯
  ├── 沿归因路径向上追溯到根因阶段
  ├── 关联具体的 Skill 和产出物质量
  └── 识别是单点问题还是系统性问题

Step 5: 输出结论
  ├── 给出明确的分析结论（主要问题类型 + 核心根因）
  ├── 量化各类问题的占比
  └── 输出优先级排序的改进建议
```

### 五、评测与自迭代机制

#### 5.1 评测流程

归因分析模块支持人工校正与自迭代，形成闭环：

AI 初始归因 → RD/PM 人工校正 → 校正结果入库 → 模型微调/规则更新 → 下次归因质量提升

#### 5.2 评测维度

| 评测维度 | 指标 | 目标值 |
| --- | --- | --- |
| 归因准确率 | 人工校正后与 AI 初始归因一致的比例 | > 75% |
| 分类覆盖率 | 所有被修改片段均能被分类的比例 | > 95% |
| 根因定位精度 | 归因到正确 Skill 的比例 | > 70% |
| 改进建议采纳率 | 改进建议被实际执行的比例 | > 50% |
| P1-d/e/f 识别准确率 | 正确区分"Spec 质量问题"与"AI 理解问题"的比例 | > 80% |

#### 5.3 人工校正规范

RD/PM 在校正归因结果时，需提供：

1. **原始分类**：AI 给出的问题分类（P1-P8 + 子类型）
2. **校正分类**：人工认为正确的分类
3. **校正理由**：简要说明为何需要校正（1-2 句话）
4. **根因补充**：如有更准确的根因描述，可补充

**人工标注适用阶段**：人工标注主要适用于归因模型冷启动阶段（前 3 个月或前 50 个需求）。当归因准确率达到目标值（> 75%）后，应逐步减少人工介入，转为抽样校正（每月抽取 10% 的归因结果进行人工复核）。

**自动化替代路径**：随着归因模型积累足够的校正样本，可通过以下方式降低人工成本：事中归因自检（4.4 节）替代部分事后人工归因；基于历史校正数据训练归因分类器；建立归因规则库（高置信度的归因模式自动应用）。

#### 5.4 自迭代触发条件

| 触发条件 | 自迭代动作 |
| --- | --- |
| 某子类型校正率 > 30% | 触发该子类型归因规则复审 |
| 新出现未覆盖的问题模式 | 触发问题分类体系扩展评审 |
| 某 Skill 归因集中度 > 40% | 触发该 Skill 专项优化 |
| 通用归因类（P7）占比 > 20% | 触发分类体系完备性评审 |
| P1-d 占比 > 20% | 触发 Spec 生成流程专项优化 |
| P8 类占比 > 15% | 触发 QA 并行链路专项优化 |

| 数据 | 来源系统 | 接入方式 | 用途 |

#### 7.2 代码片段-Skill 关联机制

归因分析的核心是"将被修改的代码片段关联到产生该代码的 Skill 执行节点"，通过 work_status.md 产出物路径作为桥梁实现。

**关联路径**：代码 diff → work_status.md 产出物路径 → Skill 执行节点

具体步骤：

1. 从 Git diff 中提取被修改的代码文件路径和行号
2. 在 work_status.md 中查找对应文件路径的产出物记录（字段：`output_files`）
3. 通过产出物记录关联到对应的 Skill 执行节点（字段：`skill_name`、`execution_id`）
4. 从 yuanxi_observability_sessions 中拉取该 Skill 执行节点的完整 context

**关联示例**：

```Plain Text
代码 diff: src/service/OrderService.java (line 120-135)
  ↓ work_status.md 查找
产出物记录: {output_file: "src/service/OrderService.java", skill: "yx-writing-code", execution_id: "xxx"}
  ↓ 关联 Skill 执行节点
归因路径: N8 编码实现 → yx-writing-code → 对应 design.md 中的 Task-003
```

#### 7.3 扩展性设计

本框架支持以下维度的扩展：

5. **新增问题类型**：在 P1-P8 基础上，可按需新增 P9+ 类型（如"多模态理解偏差"）

### 八、诊断建议

基于当前问题分类体系和数据依赖，给出以下系统性诊断建议：

1. **优先区分 Spec 质量问题与 AI 理解问题**：在数据采集阶段，应优先建立 Spec 质量评估机制，避免将"Spec 本身有问题"误归因为"AI 理解能力不足"，导致改进方向错误。
2. **建立代码片段-Skill 关联基础设施**：归因分析的前提是完整的 context 数据，建议优先打通 work_status.md 产出物路径与代码 diff 的关联链路（7.2 节机制）。
3. **从高频问题类型入手**：建议先对已有案例进行人工归因标注，识别 P1-P8 中的高频类型，优先优化对应 Skill。
4. **区分质量问题与风格问题**：P7-c（个人风格偏好）不应计入采纳率损失，建议在数据采集阶段增加修改类型标注。
5. **建立 Skill 级别的归因看板**：将归因结果按 Skill 聚合，形成各 Skill 的"问题热力图"，指导 Skill 迭代优先级。特别关注 SubAgent 节点（sub5/sub6/sub7/sub8）的归因数据。
6. **事中归因优先于事后归因**：按 4.4 节的事中归因触发点，在各 Skill 的产出物中内嵌质量自检，实现事中归因，降低事后分析成本。
7. **人工标注分阶段退出**：按 5.3 节的适用阶段说明，在归因准确率达标后逐步减少人工介入，转为抽样校正。
