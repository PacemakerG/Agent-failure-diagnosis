# AgentTrace：面向已部署多智能体系统的因果图溯源根因分析

**发表于 ICLR 2026 Workshop on Agents in the Wild**

**作者：** Zhaohui Geoffrey Wang  
USC Viterbi 工程学院 · zwang000@usc.edu

---

## 摘要

随着多智能体 AI 系统在真实场景中的大规模部署——从自动化客服到 DevOps 故障修复——故障诊断因级联效应、隐式依赖和漫长执行链路而愈发困难。本文提出 **AgentTrace**，一个轻量级因果溯源框架，用于对已部署多智能体工作流进行事后故障诊断。AgentTrace 从执行日志中重建因果图，从错误表现节点向上回溯，并使用可解释的结构特征和位置特征对候选根因进行排序——**调试阶段无需调用 LLM**。在覆盖常见部署失效模式的多样化基准测试中（550 个场景），AgentTrace 以亚秒级延迟实现高精度根因定位，显著优于启发式方法和基于 LLM 的基线。实验结果表明，因果溯源为提升真实场景下智能体系统的可靠性和可信度提供了坚实的实践基础。

## 1. 引言

由大型语言模型（LLM）驱动的多智能体系统已成为解决复杂任务的强大范式（Wu 等，2023；Hong 等，2024）。这些系统中，规划者、编码者、审核者、执行者等专职智能体通过消息传递协同完成单个智能体难以完成的目标。

在已部署的智能体系统中（如自动化客服、DevOps 故障修复、研究助手），故障往往在距根因很远的下游才浮现。当错误被发现时，多个智能体可能已基于错误假设完成了后续操作，使人工调试既慢又不可靠。这类工作流的分布式与涌现特性，意味着传统调试方法——孤立地检查单个组件——无法捕捉导致系统级故障的跨智能体因果依赖。

我们提出 **AgentTrace**，通过三个核心贡献应对上述挑战：

1. **因果图构建**：将多智能体执行过程建模为有向图，节点代表智能体行为，边捕捉信息流和因果依赖关系。
2. **后向溯源算法**：从错误表现节点出发，沿因果图向上回溯，识别所有可能相关的上游决策。
3. **故障定位实证研究**：证明基于可解释位置特征和结构特征的轻量级因果溯源，能在近似真实部署的场景下显著提升调试准确率和速度——且无需昂贵的 LLM 推理。

虽然基准场景为合成构造，但它们被设计为覆盖已部署多智能体系统中观察到的常见失效模式，例如早期规划错误逐级级联。我们在 10 个领域的 550 个失效场景上评估，AgentTrace 以亚秒延迟实现高定位准确率，显著优于启发式和基于 LLM 的基线。图 1 给出框架概览。

## 2. 相关工作

**多智能体系统。** AutoGen（Wu 等，2023）和 MetaGPT（Hong 等，2024）等框架基于推理-行动范式（Yao 等，2023）实现了复杂的多智能体协作。然而，这些系统的调试支持仍十分有限，通常依赖人工日志检查。

**AI 系统调试。** 已有 AI 可解释性工作聚焦于神经网络内部（Simonyan 等，2014；Kim 等，2018）或推理链（Wei 等，2022）。自调试方法（Chen 等，2024）用 LLM 识别错误，但需要昂贵的推理调用，且难以处理跨智能体问题。

**分布式追踪。** Jaeger（Uber，2017）和 Zipkin（Twitter，2012）等系统为微服务提供分布式追踪。我们将这些概念适配到多智能体 LLM 系统——这里智能体间的"消息"承载语义内容，而非仅仅是请求元数据。

**根因分析（RCA）。** 传统 RCA 使用统计方法（Soldani & Brogi，2022）或图算法（Page 等，1999）。最近有工作探索基于 LLM 智能体的云系统根因分析（Roy 等，2024），但这些方法并非为多智能体工作流的特定结构而设计。

## 3. AgentTrace 框架

### 3.1 问题定义

给定一条以错误状态终止的多智能体执行轨迹 $T$，目标是识别**根因节点**——即最早的决策点，修正该节点可阻止错误发生。

形式化定义：令 $G = (V, E)$ 为有向无环图，其中：
- $V = \{v_1, v_2, \ldots, v_n\}$ 代表智能体行为（工具调用、消息、决策）
- $E \subseteq V \times V$ 代表行为间的因果依赖
- $v_{\text{error}} \in V$ 是错误表现节点
- $v_{\text{root}} \in V$ 是真实根因节点

### 3.2 因果图构建

从执行日志构建因果图，识别三类边：

- **顺序边**：连接同一智能体的连续行为，捕捉智能体的推理流程。
- **通信边**：连接不同智能体之间的消息发送事件与消息接收事件。
- **数据依赖边**：通过变量引用追踪，连接数据产生行为与数据消费行为。

### 3.3 后向溯源

给定错误节点 $v_{\text{error}}$，执行广度优先后向遍历，收集指定深度内的所有祖先节点：

**算法 1：后向溯源**

```
输入：错误节点 v_error，图 G，最大深度 d
输出：候选集合 C

C ← {v_error}
frontier ← {v_error}
for i = 1 to d:
    new_frontier ← ∅
    for v in frontier:
        for u in parents(v):
            if u ∉ C:
                C ← C ∪ {u}
                new_frontier ← new_frontier ∪ {u}
    frontier ← new_frontier
return C
```

### 3.4 节点排序算法

使用五组特征的加权线性组合对候选节点进行排序：

$$\text{score}(v) = \sum_{i \in \{p, s, c, f, e\}} w_i \cdot F_i(v)$$

**表 1：完整特征列表及计算方法**

| 特征组（权重 $w_i$） | 特征 | 计算公式 | 取值范围 |
|---|---|---|---|
| **位置特征**（0.70） | 归一化位置 | $\text{pos}(v)/|V|$ | [0,1] |
| | 到错误节点距离 | $d(v, v_{\text{error}}) / \max_u d(u, v_{\text{error}})$ | [0,1] |
| | 深度比 | $\text{depth}(v) / \max_u \text{depth}(u)$ | [0,1] |
| | 反向位置 | $1 - \text{pos}(v)/|V|$ | [0,1] |
| **结构特征**（0.20） | 出度 | 归一化出度 | [0,1] |
| | 入度 | 归一化入度 | [0,1] |
| | 介数中心性 | $\sum_{s \neq v \neq t} \sigma_{st}(v)$（归一化） | [0,1] |
| | 可达性 | $|\text{descendants}(v)|/|V|$ | [0,1] |
| **内容特征**（0.05） | 错误关键词 | 内容含"error"等关键词 | {0,1} |
| | 不确定性 | 内容含"maybe"等词 | {0,1} |
| | 长度异常 | $\|len - \mu\| / (3\sigma)$（截断） | [0,1] |
| | 关键词密度 | $\text{count(keywords)} / \text{len}$ | [0,1] |
| **流程特征**（0.03） | 智能体切换 | 是否与前一节点来自不同智能体 | {0,1} |
| | 角色重要性 | 智能体角色权重 | [0,1] |
| | 通信标记 | 节点类型是否为"message" | {0,1} |
| **置信度特征**（0.02） | 明确置信度 | 若有则取置信度值，否则为 0.5 | [0,1] |
| | 对冲分数 | 对冲词数量 / 10（截断） | [0,1] |

每组分数 $F_i(v)$ 为该组内各归一化特征的均值：

$$F_i(v) = \frac{1}{|G_i|} \sum_{j \in G_i} \hat{f}_j(v), \quad \hat{f}_j(v) = \frac{f_j(v) - \min_u f_j(u)}{\max_u f_j(u) - \min_u f_j(u) + \epsilon}$$

其中 $\epsilon = 10^{-8}$ 防止除以零。权重 $w_i$ 通过在 50 个 held-out 验证场景上的网格搜索确定（见附录 B）。

## 4. 基准测试

### 4.1 场景生成

构建了包含 **550 个合成多智能体失效场景**的基准，覆盖 10 个领域（见表 2）。每个场景包含：完整执行轨迹（8–15 个智能体行为）、在特定节点系统性注入的 bug，以及根因节点的真实标注。

**表 2：基准测试领域及场景数量（共 550 个）**

| 领域 | 数量 | 领域 | 数量 |
|---|---|---|---|
| 软件开发 | 52 | 医疗协作 | 60 |
| 客户服务 | 51 | 法律文档分析 | 60 |
| 研究分析 | 51 | 教育辅导 | 60 |
| 规划与调度 | 46 | 金融咨询 | 60 |
| 金融交易 | 50 | DevOps 自动化 | 60 |

### 4.2 Bug 注入与真实标注

注入五类 bug：
- **逻辑错误**（错误条件判断，30%）
- **通信故障**（消息误解，20%）
- **数据损坏**（错误转换，20%）
- **缺失验证**（跳过校验，16%）
- **角色混淆**（超出专业范围行事，14%）

Bug 注入流程（五步）：① 从领域模板生成正确执行轨迹；② 选择 bug 位置（早期步骤 2–3：60%，中间步骤 4–6：30%，后期步骤 7+：10%）；③ 分配 bug 类型；④ 注入错误内容（如翻转比较运算符、截断消息、交换变量名）；⑤ 将级联效应传播至下游步骤。为防止方法利用显式 bug 标记，还提供了使用匿名 ID 和分离真实标注的盲测版本。

构建的因果图平均有 **10.8 个节点**（σ=2.1）和 **14.2 条边**（σ=3.5），边类型以顺序边为主（61%），其次是通信边（27%）和数据依赖边（12%）。

## 5. 实验

### 5.1 评估指标

- **Hit@k**：真实根因出现在 top-k 预测中的场景比例
- **MRR（平均倒数排名）**：$1/\text{rank}$ 的均值

### 5.2 基线方法

- **Random**：均匀随机选择节点
- **First Node**：始终选择第一个节点
- **Last Node**：选择错误节点的前一个节点
- **LLM Analysis**：使用 GPT-4 + 完整轨迹 + 提示工程（详见附录 E）

### 5.3 主要结果

表 3 显示 AgentTrace 显著优于所有基线。McNemar 检验确认所有比较均显著（$p < 10^{-10}$），与 LLM Analysis 相比的 Cohen's h = 0.77，表明有大的实际差异。

**表 3：550 场景基准上的根因定位准确率及统计显著性**（95% CI 由 B=10,000 次 bootstrap 重采样得出）

| 方法 | Hit@1 | Hit@3 | MRR | χ² | Cohen's h |
|---|---|---|---|---|---|
| Random | 9.1% | 27.3% | 0.18 | 459.2 | 2.41 |
| First Node | 3.6% | 10.9% | 0.07 | 497.0 | 2.72 |
| Last Node | 12.7% | 38.2% | 0.25 | 436.5 | 2.29 |
| LLM Analysis | 68.5% | 81.4% | 0.74 | 102.5 | 0.77 |
| **AgentTrace** | **94.9%** [92.9, 96.7] | **98.4%** [96.9, 99.4] | **0.97** | – | – |

### 5.6 案例展示

**案例 1：软件开发** — 任务：实现均值计算

```
步骤 1 [规划者]：分析需求 → "所有数求和，计数，相除"
步骤 2 [编码者]：写实现 → "def average(nums): ..."
步骤 3 [编码者]：添加边界处理 【BUG】
    → "if nums = []: # BUG：= 应为 =="
步骤 4 [审核者]：审查代码 → "代码看起来正确。"
步骤 5 [执行者]：运行测试 【ERROR】
    → "SyntaxError: 第 2 行语法错误"
```

根因：步骤 3（逻辑错误）。AgentTrace 正确识别步骤 3；LLM 基线选择步骤 5（错误节点），体现了其倾向于选择错误表现而非向上追溯的局限性。

**案例 2：研究分析** — 任务：分析 Transformer 架构相关论文

```
步骤 1 [搜索者]：查询数据库 → "150 篇论文"
步骤 2 [搜索者]：过滤 → "25 篇相关论文"
步骤 3 [分析者]：提取发现 【BUG】
    → "Attention 正被 MLP 取代"（实际：Attention 被增强）
步骤 4 [综合者]："领域正在远离基于 Attention 的模型……"
步骤 5 [写作者]："Transformer 正走向过时"
步骤 6 [写作者]：完成报告 【ERROR】
    → "与最新 SOTA 结果相矛盾"
```

根因：步骤 3（通信故障）。AgentTrace 正确识别步骤 3；LLM 选择步骤 4（传播步骤），说明区分根因与下游效果的困难。

### 5.7 运行时性能

AgentTrace 平均处理一条轨迹耗时 **0.12 秒**，而基于 LLM 的分析需 8.3 秒。**69 倍的速度提升**使交互式调试工作流成为可能。

## 6. 讨论

**为什么位置特征占主导？** 位置信号的主导地位不应仅被解读为基准测试的人工制品。在许多已部署的智能体系统中，早期规划或路由决策决定了整个下游执行，使得早期错误的影响极为突出。我们的分析揭示：在执行轨迹中出现越早的 bug，越容易造成晚期才表现出来的错误——这反映了分层多智能体工作流的基本属性：上游决策约束下游行为。

**失败案例分析。** 在 28 个（5.1%）AgentTrace 未能将正确根因排在首位的场景中：
- **多个可信根因**（12 例，42.9%）：例如步骤 2（错误搜索查询）和步骤 3（过于严格的过滤）都对错误有贡献。
- **非常规位置的 bug**（8 例，28.6%）：后期 bug（步骤 7+）的特征分布与多数样本不同，导致 AgentTrace 过度优先选择早期步骤。
- 其余失败来自非典型因果结构（5 例）和特征提取问题（3 例）。

潜在改进方向：位置自适应权重和多根因检测。

**局限性。** 当前评估聚焦于单根因的合成场景。真实多智能体故障通常涉及多个共同因素和更复杂的因果结构。此外，基准假设执行日志准确无误，而生产系统中这一点未必成立。

**对智能体安全性的启示。** 随着多智能体系统被部署于高风险领域，快速识别和理解故障的能力对维护信任与安全至关重要。AgentTrace 为事后分析提供了基础，可为即时修复和系统性智能体设计改进提供支撑。

---

## 7. 结论

本文提出 AgentTrace，一个基于因果图的已部署多智能体系统根因定位框架。该方法在多样化基准上实现高准确率，同时保持亚秒级延迟。框架依赖可解释的结构特征和位置特征——而非昂贵的 LLM 推理——使其在生产环境的交互式调试中切实可行。未来工作将扩展至处理多并发根因，并在生产轨迹上进行验证。我们希望 AgentTrace 能成为在真实、可靠性关键环境中部署的智能体系统的实用诊断层。

## 附录 B：特征权重学习

特征组权重通过在 50 个 held-out 验证场景上的网格搜索确定：

$$w^* = \arg\max_w \sum_{i=1}^{50} \mathbb{1}[\text{rank}(v_{\text{root}} | w) = 1]$$

搜索空间：$w_p \in \{0.5, 0.6, 0.7, 0.8\}$，$w_s \in \{0.1, 0.15, 0.2, 0.25\}$，$w_c \in \{0.03, 0.05, 0.07, 0.1\}$，$w_f \in \{0.02, 0.03, 0.05\}$，$w_e \in \{0.01, 0.02, 0.03\}$，约束条件 $\sum_i w_i = 1$。

## 附录 C：统计分析细节

**Bootstrap 置信区间**：使用 B=10,000 次 bootstrap 重采样计算 95% CI，取第 2.5 和第 97.5 百分位数为区间边界。

**McNemar 检验**：构建 2×2 一致/不一致预测列联表，检验统计量为 $\chi^2 = (|n_{01} - n_{10}| - 1)^2 / (n_{01} + n_{10})$，服从自由度为 1 的 $\chi^2$ 分布。

**效应量**：使用 Cohen's h = $2\arcsin(\sqrt{p_1}) - 2\arcsin(\sqrt{p_2})$，其中 |h| < 0.2 为小效应，0.2 ≤ |h| < 0.5 为中效应，|h| ≥ 0.5 为大效应。
