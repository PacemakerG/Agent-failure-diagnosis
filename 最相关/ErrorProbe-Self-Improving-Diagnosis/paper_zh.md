## 摘要

基于大型语言模型（LLM）的多智能体系统（MAS）能够支持复杂问题求解，但也带来了严峻的调试挑战：长交互轨迹、智能体间依赖关系，以及延迟的错误表现。现有诊断方法通常依赖昂贵的专家标注或"LLM 即裁判"范式，难以在长上下文中精确定位决定性错误步骤。

本文提出 **ErrorProbe**，一个用于语义故障归因的自改进框架，可识别责任智能体及错误起源步骤。框架通过三阶段流水线运作：① 将 MAS 故障分类体系操作化，检测局部异常；② 执行症状驱动的后向溯源以剪枝无关上下文；③ 由专职多智能体团队（策略者、调查者、仲裁者）通过工具接地执行来验证错误假设。关键在于，**ErrorProbe 维护一个经验证的情节记忆，仅在错误模式被可执行证据确认时才更新，无需人工标注**。

在 TracerTraj 和 Who&When 基准上的实验表明，ErrorProbe 显著优于基线，尤其在步骤级定位上；经验证的记忆还能实现跨领域迁移，无需重新训练。

## 1. 引言

基于 LLM 的多智能体系统（MAS）在软件工程、网页导航、科学推理、教育和工具使用等多样化领域展现出强劲性能。其核心吸引力在于：通过将复杂任务分解为多个步骤并分配专职角色（如架构师、工程师、测试员），MAS 能够解决单次 LLM 调用难以完成的问题。

然而，MAS 的能力提升也带来了实际调试问题。考虑一个典型故障模式：架构师引入了细微的规格错误，工程师忠实地实现了它，测试员验证了错误行为。运行可能在很多轮之后才失败——此时多个智能体已基于最初的错误构建了后续操作。面对这种情况，从业者需要回答核心诊断问题：**"哪个智能体导致了故障，错误在哪一步起源？"**

本文研究 MAS 中的故障归因问题：给定多智能体交互轨迹和故障症状（如最终答案错误、异常或未满足的约束），目标是识别责任智能体和起源步骤。这一问题在多智能体场景下具有独特挑战：

- **长交互轨迹**：生产级 MAS 运行可跨越数十至上百轮，往往超出实际上下文窗口。
- **延迟表现**：早期错误可能很晚才浮现，使因果归因复杂化。
- **智能体间依赖**：每个智能体以前序智能体的输出为条件，形成长（有时有分支的）因果链。
- **多样化故障模式**：故障涵盖规格错误、协调失败和验证缺口，每种模式需要不同的诊断方式。

现有工作主要分三类：① **分类体系驱动研究**（如 MAST）提供了系统的故障模式视角，但构建和标注需要大量专家标注，难以扩展；② **监督式专用追踪器**性能较好，但依赖昂贵的数据生成流水线，且随智能体框架演进可能需要持续重训；③ **将通用 LLM 用作裁判**开销低、适用广，但对于长延迟失败轨迹中的决定性错误步骤定位，可靠性仍不足。

这些局限促使我们构建**自进化多智能体诊断框架**：通过将诊断分解为后向溯源、假设生成和工具接地验证的专职角色，管理长交互轨迹的复杂性，在不牺牲通用性的前提下提升归因准确率。

**本文贡献：**

1. 提出 **ErrorProbe**，一个用于 LLM 多智能体系统语义故障归因的框架，可定位责任智能体和错误起源步骤。
2. 将 MAST 分类体系操作化为轻量级检测器，扫描交互轨迹中的局部异常（如未对齐或验证缺口），为后续推理提供可解释的结构化故障模式信号。
3. 通过经验证的记忆实现有效的跨领域迁移，克服了先前方法常需重训或难以应对分布偏移的脆弱性。

## 2. 相关工作

**多智能体错误归因。** 越来越多的工作致力于多智能体系统中的故障归因。Zhang 等（2025b）形式化了这一问题，指出步骤级定位的困难。后续方法探索了频谱式分析、层级上下文分解和通信效率优化。互补研究通过分类体系、根因诊断基准和人工标注调试数据集来刻画故障。与依赖大量监督、昂贵回放或领域敏感缓存的先前方法不同，本文专注于将后向溯源与结构化故障模式信号相结合，对长延迟失败轨迹进行有效的语义诊断。

**LLM 与智能体评估。** "LLM 即裁判"范式已成为可扩展评估的标准，通常通过思维链或明确评分细则来增强。近期研究将其扩展为"智能体即裁判"，为智能体工作流提供过程级反馈。然而，标准裁判通常输出标量或成对排名，我们的场景需要因果定位，因此需超越自由式评判，转向辅以语义验证的轨迹级后向推理。

**自进化智能体与记忆。** 自改进智能体利用语言强化和记忆：迭代反思或情节缓冲区来精化输出；技能库和记忆管理架构支持终身学习。自进化框架在测试时集成这些组件以适应智能体行为。针对这些系统中的记忆污染挑战，我们提出**写前验证更新门**，确保只有得到验证信号支持的补丁才被提交，在防止漂移的同时保留可复用的错误模式。

## 3. 问题形式化

**多智能体故障归因**被形式化为序列预测任务。令 $T$ 为多智能体系统生成的交互轨迹空间。轨迹由按时间顺序排列的基于轮次的交互构成，每步记录发送者身份（智能体）、接收者和语义内容（自然语言消息、代码块或工具执行输出）。

任务实例定义为元组 $(x, y)$，其中 $x \in T$ 是包含消息和工具输出序列 $x = [m_1, m_2, \ldots, m_L]$ 的失败执行轨迹，$y = (a^*, t^*, f^*)$ 是真实归因元组：

- $a^*$：根因对应的**责任智能体**
- $t^* \in [1, L]$：**决定性错误步骤**——逻辑故障首次引入的具体交互（如幻觉指令、错误参数、有 bug 的代码片段），而非故障症状变得可见之处
- $f^*$：**故障模式**，采用 MAST 分类体系（Cemri 等，2025），将多智能体故障组织为三个层级族系下的 14 种错误模式：
  - **规格问题（FC1）**：初始设置或提示中的错误（如违反任务约束的"不遵守任务规格"，或智能体超出分配角色的"不遵守角色规格"）
  - **未对齐（FC2）**：智能体间协调失败（如"忽略其他智能体输入"导致轮次间上下文丢失，或"推理-行动不匹配"导致输出与智能体内部思维过程相矛盾）
  - **验证失败（FC3）**：质量保证流程中的缺陷（如"无验证/不完整验证"——测试者未经测试即通过，或"错误验证"——有效解被拒绝）

标准评估将其视为 i.i.d. 分类问题。然而，真实调试是持续过程，诊断系统应从过往故障中学习。因此我们定义**序列诊断**设定：系统维护可变记忆状态 $M_i$，给定故障任务流 $\{(x_i, y_i)\}_{i=1}^N$，模型预测 $\hat{y}_i = F(x_i; M_{i-1})$ 并更新状态：

$$M_i \leftarrow \text{Update}(M_{i-1}, x_i, \hat{y}_i, \text{Verify}(\hat{y}_i))$$

目标是在最小化记忆大小和误更新的同时，最大化流上的累积归因准确率。

## 4. ErrorProbe 框架

ErrorProbe 通过三阶段流水线运作：① 基于 MAST 分类体系的结构分解，② 症状驱动的后向溯源以剪枝无关上下文，③ 带验证记忆的自进化多智能体诊断循环。整体架构见图 2。

### 4.1 MAST 引导的结构分解

原始交互轨迹通常是非结构化且充满噪声的。为提供语义锚定，我们将 MAST 分类体系操作化为轻量级信号检测器。

我们先将轨迹解析为结构化表示 $S_x = \{(\text{agent}_t, \text{role}_t, \text{action}_t)\}_{t=1}^L$：
- **Role**（角色）：从系统提示中提取的分配角色（如架构师、软件工程师、产品经理）
- **Action**（行为）：轮次的语义类型（如自然语言指令、工具调用或执行输出）

然后对每个节点应用分类体系条件提示，标记潜在的局部异常。与作为输入的全局故障症状（表明任务失败）不同，这些标签识别特定步骤级偏差发生的位置（如"步骤 5 处工具输出被忽略"的警告）。这些弱信号作为后续推理智能体的启发式先验，将搜索空间从 $L$ 步缩减为更小的候选区域。

### 4.2 症状驱动的后向溯源

MAS 调试的核心挑战在于根因（如步骤 $t=5$ 处的错误参数）与症状（如步骤 $t=50$ 处的崩溃）之间的距离。处理完整线性历史往往导致"迷失在中间"现象。

为此，我们实现**后向溯源**机制来重建故障因果链。给定步骤 $L$ 处的故障症状：

1. **依赖解析**：构建依赖图 $G = (V, E)$，节点为消息，边代表信息流（如智能体 B 引用了智能体 A 的输出）。
2. **递归剪枝**：从症状节点 $v_L$ 出发，对入边执行广度优先搜索，识别错误的**有效感受野**。
3. **上下文压缩**：屏蔽未连接的分支（如已成功的并行子任务），仅保留因果谱系 $x' \subset x$。

压缩后的上下文 $x'$ 将模型的注意力聚焦于实际影响故障的智能体和步骤，显著降低诊断智能体所面对的噪声。

### 4.3 多智能体诊断架构

诊断由三个专职智能体组成的团队执行，模拟人类调试工作流：假设生成、验证和裁决。

**策略者（评估负责人）**：高层规划者。接收压缩轨迹 $x'$ 和来自 4.1 节的结构标签，目标是制定一组假设 $H = \{(s_j, \text{suspected\_mode})\}$。为此，策略者查询长期记忆 $M$ 以检索 $k$ 个历史故障模式。在"冷启动"场景（记忆为空）或未找到语义相似模式时，策略者优雅降级为基于第一原理推理，仅依赖结构化 MAST 标签和轨迹上下文。随着系统遇到并验证新故障，记忆自然填充，策略者逐步从零样本推理过渡到模式感知诊断。

**调查者（Bug 检查员）**：为对抗错误原因的幻觉，调查者严格基于工具使用。对每个假设 $h \in H$，必须产生证据 $E_h$。专用工具集包括：
- **CodeExec**：在沙箱中重新运行代码片段，确认运行时错误。
- **LogicProbe**：验证特定前置/后置条件（如"检查步骤 $t$ 中的变量 $X$ 是否符合步骤 $t-5$ 中的约束"）。

调查者不能简单声称"代码有错"，必须生成差异对比或执行日志来证明不一致。

**仲裁者（验证者）**：最终决策门。聚合假设及其对应证据集 $\{(h, E_h)\}$，过滤掉证据为空或不确定的假设，输出最终预测 $\hat{y} = (a, t, f)$ 及置信度分数 $c \in [0, 1]$。仲裁者还决定所诊断的模式是否足够新颖且稳健以提交到记忆，充当防止记忆污染的保障。

### 4.4 经验证的情节记忆

**验证门**：仲裁者对当前任务作出裁决，而验证门作为长期保留的更严格过滤器，确保只有具有高置信度、可重现工具证据的诊断才被提交到记忆。形式上，新诊断 $\hat{y}_t$ 及其关联证据 $E_t$ 当且仅当 $\text{Verify}(E_t) \wedge c_t > \tau$ 时提交到记忆状态 $M_t$：

$$M_t = M_{t-1} \cup \{(\hat{y}_t, E_t, \sigma_t)\}$$

这防止了"幻觉"错误模式污染诊断历史。

**经验证的情节记忆**：ErrorProbe 维护动态记忆 $M$，每个条目 $e \in M$ 存储为元组 $\langle s_e, v_e, \sigma_e \rangle$：
- **结构签名** $s_e$：包含检测到的 MAST 异常（如 `step_repetition`）、涉及的工具或 API 端点、抽象参数类型和上下文槽（如智能体角色或任务领域）
- **诊断配方** $v_e$：包含典型错误模式和验证守卫
- **元统计** $\sigma_e$：跟踪时效性、频率、影响和性能增量

**相关性评分与检索**：诊断新轨迹 $x$ 时，构建查询签名 $s_x$ 并计算综合结构相似性与质量加权 RFI-Δ 分数的相关性得分：

$$S(x, e) = \underbrace{\text{Sim}(s_x, s_e)}_{\text{结构匹配}} \times \underbrace{S_{\text{RFI}}(e)}_{\text{质量分数}}$$

其中 $\text{Sim}(\cdot)$ 在签名字段上计算加权匹配（要求 MAST 标签精确匹配），$S_{\text{RFI}}$ 综合时效性、频率、影响和性能增量。为解决冷启动问题，设定严格相似度阈值 $\tau_{\text{ret}} = 0.75$；若 $\max_e \text{Sim}(x, e) < \tau_{\text{ret}}$，检索返回空集，分析器回退到仅使用 MAST 分类体系的第一原理推理。

**算法 1：自进化诊断流程**

```
输入：失败轨迹 x_t，记忆 M_{t-1}
输出：诊断 ŷ，更新后的记忆 M_t

1. 压缩：x'_t ← BackwardTrace(x_t, MastPriors)
2. 检索：P ← Retrieve(M_{t-1}, x'_t, k)
3. 规划：H ← Strategist(x'_t, P)
4. for 每个假设 h in H:
5.     E_h ← Investigator(h, Tools)
6. 裁决：(ŷ, c, E) ← Arbiter(H, {E_h})
7. if c ≥ τ 且 Verify(E):
8.     M_t ← Update(M_{t-1}, ŷ, E)
9. else:
10.    M_t ← M_{t-1}
11. return ŷ
```

## 5. 实验

### 5.1 实验设置

**任务与指标。** 评估两项预测任务：① **智能体级归因**——识别责任智能体 $a^*$；② **步骤级归因**——定位决定性错误步骤 $t^*$。报告各数据集的 Top-1 智能体准确率和步骤准确率，以及三个基准的宏平均。

**基准数据集。**
- **TracerTraj**：基于 Zhang 等（2026）的代码生成子集，具有长上下文轨迹，包含程序化注入的故障，提供精确的合成真实标注。涵盖 MetaGPT、OpenHands、AutoGen 等主流框架。
- **Who&When-Algo**：来自 Zhang 等（2025b）的真实故障数据集，采集自 CaptainAgent，智能体团队由算法动态生成。
- **Who&When-Hand**：来自 MagenticOne 的真实故障，代表手工设计的智能体团队，涉及开放式网页导航等复杂真实场景，真实标注由人工严格标注。

**对比方法（均使用 Claude 3.7 Sonnet、GPT-OSS-120B 和 Qwen3 32B 作为骨干）：**
- **LLM 即裁判**：将完整轨迹和故障描述输入单个 LLM 直接输出归因结果
- **智能体即裁判（基线）**：来自 Zhuge 等（2025）的工具增强框架，评估者是配备文件读取、代码定位、依赖图等能力的自主智能体
- **智能体即裁判（我们的）**：完整 ErrorProbe 流水线但不含验证情节记忆模块
- **ErrorProbe（含记忆）**：完整框架

**关键发现：**

1. **单次判断在长轨迹上的步骤定位能力不足**：LLM 即裁判在智能体归因上达到中等准确率，但步骤级准确率显著偏低，在 TracerTraj 上 Step 准确率低于 10%。单次 LLM 遍历长轨迹倾向于过度关注后期症状，欠利用早期因果证据。

2. **工具增强的智能体评估有帮助，但对协议敏感**：智能体即裁判基线在多个场景中改善了步骤级归因（如 Who&When-Algo Step 对 Claude 提升 6.3%），说明交互式检查能发现原始轨迹之外的额外证据。但智能体级归因在该基线下有时会退化，表明需要更结构化的诊断流程。

3. **ErrorProbe 大幅改善了决定性步骤定位**：引入 MAST 引导的分解和后向溯源后，步骤定位在所有三个基准上均获得大幅提升——Claude 步骤准确率平均提升 20.6 个百分点，GPT-OSS 提升 13.7 个百分点。最大提升出现在 TracerTraj 上，与 ErrorProbe 从长轨迹中恢复早期因果步骤的目标一致。

4. **验证记忆提供额外增益，对弱骨干尤为明显**：启用验证情节记忆后性能进一步提升，GPT-OSS 的提升最为明显，Claude 在步骤准确率上也有稳定改善（TracerTraj Step +4.8%，Who&When-Algo Step +0.8%）。

## 6. 记忆分析

### 6.1 域内记忆评估

在四个领域对比"无记忆基线"与"记忆增强变体"：MBPP 和 KodCode（代码）、GSM8K 和 MATH（数学推理）。由于收集带标注的多智能体故障代价高昂，我们使用多种 MAS 框架生成轨迹，保留失败轨迹进行分析，并手工标注测试集。

**表 2：域内记忆对比**

| 设置 | MBPP<br>Agent/Step | KodCode<br>Agent/Step | GSM8K<br>Agent/Step | MATH<br>Agent/Step |
|---|---|---|---|---|
| ErrorProbe | 87.3% / 33.3% | 64.8% / 47.2% | 25.0% / 30.0% | 68.6% / 80.0% |
| ErrorProbe + 记忆 | 88.9% / 36.5% | 81.9% / 56.8% | 50.0% / 65.0% | 68.6% / 85.7% |
| **提升** | +1.6 / +3.2 | +17.1 / +9.6 | +25.0 / +35.0 | +0.0 / +5.7 |

四个领域平均提升：智能体归因 **+10.9 分**，步骤定位 **+13.4 分**。

提升幅度与领域内错误模式的重复性相关：GSM8K 步骤提升最大（+35.0），与其故障常来自重复算术或逻辑模板一致；KodCode 智能体提升显著（+17.1），反映循环出现的协调和工具/API 误用模式；MATH 提升较小，符合其推理链更加异质的特点。

### 6.2 记忆扩展性

随着 KodCode 训练轨迹数量增加，智能体准确率提升 1.6%，而步骤准确率更显著提升 4.0%。这表明基础 LLM 对智能体归因已有较强先验，但难以精确定位决定性错误步骤；记忆模块提供了"错误签名"来消除根因步骤的歧义。

在域外（OOD）设置中（将 KodCode 学到的模式应用于 TracerTraj），也观察到稳定的上升趋势，表明写前验证门成功过滤了领域特定噪声，只保留了跨框架可泛化的结构性模式。

### 6.3 案例分析：记忆将弃权转变为正确归因

**场景**：Shell Sort 实现任务。真实标注：智能体=工程师，步骤=8。

```
"工程师声称成功实现了 Shell Sort，但文件内容被截断。"

基线预测：智能体=unknown，步骤=[10] → 弃权（错误）
含记忆预测：智能体=工程师，步骤=[8] → 正确归因（金标准步骤 8）
```

基线因明确证据不足而弃权。但含记忆系统检索到一个守卫，建议"验证所有文件操作是否成功执行"，这使模型识别出 `reasoning_action_mismatch` 模式，正确归因于未经验证的声明。

## 7. 结论

本文提出 ErrorProbe，旨在解决复杂多智能体工作流中语义故障归因的挑战。通过将症状驱动的后向溯源与经验证的情节记忆综合，有效过滤交互噪声，利用历史错误模式提升诊断精度。在三个多样化基准上的实证评估确认，ErrorProbe 持续优于近期基线，在时间错误定位这一难任务上尤为突出。这些结果表明，结构化的工具接地验证是构建能够自我改进的可信赖多智能体系统的必要组成部分。

---

## 局限性

**依赖显式信号。** 症状驱动的后向溯源以可检测异常的存在为前提。**"静默故障"**——智能体产生技术上有效但语义错误的输出而不触发 MAST 检测器——仍是挑战。

**推理延迟。** 多智能体诊断团队（策略者、调查者、仲裁者）的可靠性以增加推理计算为代价。工具辅助验证的迭代性质目前限制了其在超低延迟生产环境中的适用性。我们认为这是高精度离线或异步调试工作流中必要的权衡。

**骨干模型多样性。** 由于多智能体评估计算成本高，仅限定了三个模型系列。未来工作应扩展到更广泛的 LLM 骨干以确认架构无关的泛化性。

## 附录 B：扩展实验

### B.1 Who&When 协议基线

在 Who&When 基准上对比三种归因协议：**一次性（all at once）**、**逐步（step by step）**、**二分搜索（binary search）**。

**关键发现：**
- 在 Who&When-Algo 上，一次性协议在智能体归因上最强（57.94–61.11%），当轨迹结构较规律、故障信号局部显著时，单次判断可以成功。
- 手工设计轨迹受益于迭代细化：一次性判断步骤定位基本失败（1.72–5.17%），逐步和二分搜索均大幅改善。
- TracerTraj-Code 是步骤定位最难的数据集，二分搜索最有效（8.66–16.54%），而一次性和逐步方法步骤准确率极低（2.36–4.72%）。
- **协议选择的方差大于模型选择的方差**，尤其在步骤定位上。
