# XAI 用于编码智能体故障分析：将原始执行轨迹转化为可操作洞察

**作者：** Arun Joshi（独立研究者，Islington College 校友，尼泊尔）  
arXiv:2603.05941v1 [cs.SE] 2026年3月6日

---

## 摘要

基于大型语言模型（LLM）的编码智能体在自动化软件开发任务中展现出广阔前景，但它们频繁以难以理解和调试的方式失败。尽管通用 LLM（如 GPT）能对故障提供临时性解释，但原始执行轨迹即使对经验丰富的开发者也难以解读。

本文提出一种系统性的可解释 AI（XAI）方法，将原始智能体执行轨迹转化为结构化、人类可解读的解释。该方法由三个关键组件构成：① 从真实智能体故障分析中衍生的领域专用故障分类体系；② 使用定义好的标注模式对故障进行自动分类的系统；③ 生成可视执行流、自然语言解释和可操作建议的混合解释生成器。

通过对 20 名参与者（10 名技术人员、10 名非技术人员）的用户研究，我们证明该方法使用户识别故障根因的速度提升 **2.8 倍**，提出正确修复方案的准确率提升 **73%**（相比原始执行轨迹）。我们的结构化方法通过提供一致的领域专用洞察及集成可视化，优于临时调用最先进模型的解释方案。本工作建立了系统性智能体故障分析框架，回应了软件开发工作流中对可解释 AI 系统的迫切需求。

**关键词：** 可解释 AI、LLM 智能体、编码智能体、故障分析、人机交互

## 1. 引言

LLM 越来越多地被部署为能够生成、调试和测试软件的自主编码智能体。GitHub Copilot、AutoGPT、基于 LangChain 的编码助手等工具承诺通过自动化常规编程任务来革新软件开发。然而，这些智能体频繁以微妙而复杂的方式失败——生成错误代码、误解需求或陷入无效循环。

当智能体失败时，开发者面对的是包含嵌套工具调用、中间推理步骤和错误消息的原始执行轨迹，往往绵延数百行日志。即使对经验丰富的开发者，从这些轨迹中理解智能体为何失败也极具挑战；对于需要就智能体部署和可靠性作出决策的非技术利益相关者而言，则几乎不可能。

### 1.1 理解智能体故障的挑战

考虑一个试图解决编程问题的编码智能体，典型的故障轨迹可能包含：
- 原始任务描述
- 智能体内部推理（通过思维链）
- 多次工具调用（代码执行、验证、调试）
- 代码执行失败的错误消息
- 智能体对错误的响应（或缺乏响应）
- 系统级元数据（迭代次数、超时、资源限制）

人们可能以为直接让通用 LLM（如 ChatGPT）解释这些轨迹就够了。但我们的预实验揭示了四个显著局限：

- **局限 1：不一致性。** 临时 LLM 解释在质量和关注点上差异显著，使系统性调试困难。
- **局限 2：缺乏结构。** 通用 LLM 不具备关于常见智能体故障模式的领域专用知识。
- **局限 3：缺失视觉上下文。** 纯文本解释无法传达对理解智能体行为至关重要的执行流和决策点。
- **局限 4：无可操作指导。** 通用解释很少提供针对性的、经过验证的修复建议。

### 1.2 我们的方法：结构化 XAI 用于智能体故障

我们提出一种将原始执行轨迹转化为结构化、可操作解释的系统性方法。核心洞察是：**智能体故障遵循可预测的模式，可以被分类化、自动分类，并借助领域专用知识加以解释**。

**主要贡献：**
1. **故障分类体系**：首个面向编码智能体的综合故障分类体系，来自对 32 个真实故障的系统性分析。
2. **自动分类**：基于 GPT-4 结构化输出（函数调用）的系统，自动分类故障准确率达 **82%**，无需人工标注。
3. **混合解释系统**：集成 XAI 流水线，生成可视执行流、自然语言解释、反事实分析和可操作建议。
4. **实证验证**：用户研究（N=20）证明该方法在多项指标上优于原始轨迹和基于 GPT 的解释。

## 2. 相关工作

### 2.1 基于 LLM 的编码智能体

LLM 的近期进展使自主编码智能体得以实现。Chen 等引入了 HumanEval 基准用于评估代码生成；Austin 等展示了程序综合能力；LangChain 和 AutoGPT 等现代框架普及了将 LLM 与工具使用相结合的智能体架构。然而，这些智能体面临严峻的可靠性挑战——Zhang 等将规划失败确定为主要瓶颈，近期关于智能体调试的工作也开始对故障模式进行分类化。

### 2.2 可解释 AI（XAI）

XAI 领域已发展出众多解释模型行为的技术：LIME 和 SHAP 提供单个预测的局部解释；注意力可视化和显著图为神经网络决策提供洞察。对于语言模型，思维链提示既改善性能也增强可解释性。然而，这些技术聚焦于单个模型输出，而非多步智能体行为。

### 2.3 智能体可观测性与调试

LangSmith 等可观测性平台为智能体执行提供追踪能力；Wang 等提出了多智能体系统的调试技术。本工作的差异在于：① 专注于编码智能体而非通用智能体；② 提供从原始轨迹到解释的完整端到端系统；③ 通过人类评估验证方法，对比原始轨迹和通用 LLM 解释两个基线。

### 2.4 人机交互与开发者工具

HCI 研究长期强调可理解错误消息和调试支持的重要性。针对 AI 辅助编程，近期工作探索了开发者如何与代码生成工具交互，以及他们认为哪类解释有用。本文贡献在于证明：即使两者都使用最先进的 LLM，专业化的结构化解释仍优于通用方法。

## 3. 方法

### 3.1 智能体实现与数据收集

使用 LangChain 和 GPT-4 实现了一个编码智能体，用于解决 HumanEval 基准中的编程问题。智能体架构遵循标准 ReAct 模式，可访问代码执行和验证工具。

为系统性探索故障模式，设计了四个维度的实验场景：
- **迭代限制**：最多 1、2、5 或 10 次迭代
- **提示质量**：最简、基础、详细或全面
- **工具可用性**：完整、有限或最少工具集
- **任务难度**：简单、中等或困难问题

从 87 次智能体运行中收集执行轨迹，得到 **32 次失败**和 55 次成功。每条轨迹包含：完整消息历史（人类提示、智能体响应、工具调用）、执行元数据（时间、迭代次数、场景配置）、错误信息（类型、消息、堆栈跟踪）和最终结果（成功/失败、产生的输出）。

**关键发现：迭代精化失败**占所有失败的 **56%**，"超过迭代限制而未取得进展"是最常见模式。这表明改善错误恢复机制能显著提升智能体可靠性。

### 3.2 故障分类体系

**表 1：编码智能体故障分类体系**

| 类别 | 子类别 | 数量 |
|---|---|---|
| 规划失败 | 问题分解错误 | 1 |
| 代码生成失败 | 逻辑错误（实现错误） | 2 |
| 测试/验证失败 | 未运行验证测试 | 2 |
| 理解失败 | 误解问题需求 | 9 |
| 迭代精化失败 | 超过迭代限制而未取得进展 | 18 |

### 3.3 自动故障分类

为实现实际部署，我们开发了使用 GPT-4 通过函数调用实现结构化输出的自动标注系统。分类器：
1. 从原始轨迹提取关键特征（错误消息、迭代次数、执行模式）
2. 将这些特征连同分类体系一起呈现
3. 提示 GPT-4.1 对故障分类并提供置信度分数
4. 返回包含类别、子类别和推理的结构化标注

**评估**（在 32 个失败上手工验证）：
- 整体准确率：**82.1%**（26/32 正确）
- 高置信度预测（p > 0.8）：**90.5%** 准确率（21 个中 19 个正确）
- 与人工标注的 Cohen's κ：**0.76**（实质性一致）

误分类主要发生在多种故障模式同时存在的模糊情况。对于高风险应用，系统可对低置信度预测标记以供人工审查。

## 4. XAI 系统架构

系统通过三个集成组件将原始执行轨迹转化为综合解释。

### 4.1 组件一：执行流可视化

可视化组件生成表示智能体执行过程的有向图：

**节点**代表不同的执行步骤：
- 智能体推理/规划步骤
- 工具调用（代码执行、验证）
- 错误发生点
- 决策点

**边**展示智能体的执行流。

可视化使用 Graphviz 生成，导出为 PNG（用于嵌入）和 SVG（用于交互查看）两种格式。关键在于，这些可视化突出故障点，并展示导致故障的执行上下文——这类信息仅靠文字难以传达。

### 4.2 组件二：自然语言解释

解释生成器使用 GPT-4 生成结构化自然语言描述：

- **根因分析**：结合分类体系分类和执行上下文，识别故障的根本原因。
- **故障机制**：解释故障在具体执行中如何表现，包括相关中间步骤和决策点。
- **上下文整合**：纳入场景元数据（迭代限制、提示质量），解释故障是源于智能体自身局限还是配置约束。

### 4.3 组件三：建议引擎

建议组件提供与故障类别对应的可操作指导：

- **反事实分析**：描述能导致成功的最小变更，帮助用户理解故障边界。
- **即时修复**：可立即实施的具体变更——配置调整（迭代限制、超时）、提示工程改进、工具可用性修改。
- **长期改进**：持续可靠性的战略性增强——在领域专用数据上微调、架构修改、评估框架增强。
- **上下文专用指导**：针对具体故障实例定制的建议，考虑任务难度和领域、当前配置、故障严重性和频率、资源约束。

### 4.4 输出格式

系统生成三种输出格式：
- **HTML 报告**：包含嵌入可视化、解释和建议的综合文档，面向技术和非技术受众。
- **JSON**：用于编程访问的结构化数据，可集成至 CI/CD 流水线、监控系统和自动分析工具。

## 5. 评估

### 5.1 用户研究设计

**参与者**：共 20 名，分为两组：
- **技术参与者**（n=10）：有 0–5 年专业经验的软件开发者
- **非技术参与者**（n=10）：产品管理、设计、标注团队等非工程岗位的专业人员

**研究协议**：参与者完成三项任务：

- **任务 1——故障理解**：在三种条件之一下（被试内设计，顺序随机化），回答"出了什么问题？（根因识别）""为什么会发生？（故障机制）""如何修复？（解决方案提议）"
- **任务 2——修复提案**：提出防止故障的具体变更，由两位独立专家评审员评估正确性和完整性
- **任务 3——置信度与偏好**：评分对理解的置信度（1-7 李克特量表）并对条件排名

**三种条件：**
1. **原始轨迹**：包含所有消息、错误和元数据的完整执行轨迹
2. **LLM 解释**：原始轨迹 + 任意通用 LLM 的解释（提示语："解释这个智能体为何失败以及如何修复"）
3. **我们的 XAI**：完整系统输出（可视化 + 解释 + 建议）

每位参与者查看 3 个不同智能体故障（每种条件一个），选自不同分类体系类别和难度级别。

### 5.2 定量结果

**表 2：用户研究结果（均值 ± 标准差）**

| 指标 | 原始轨迹 | 通用 LLM | 我们的 XAI |
|---|---|---|---|
| **技术参与者（n=10）** | | | |
| 理解时间（分钟） | 8.4±2.1 | 5.2±1.3 | **3.0±0.8\*\*** |
| 根因准确率（%） | 42±15 | 68±12 | **89±8\*\*** |
| 修复质量（1-5） | 2.6±0.8 | 3.4±0.6 | **4.3±0.5\*\*** |
| 置信度（1-7） | 3.2±1.1 | 4.8±0.9 | **6.1±0.7\*\*** |
| **非技术参与者（n=10）** | | | |
| 理解时间（分钟） | 12.8±3.2 | 7.1±1.8 | **4.2±1.1\*\*** |
| 根因准确率（%） | 18±12 | 52±18 | **76±11\*\*** |
| 修复质量（1-5） | 1.4±0.6 | 2.8±0.7 | **3.8±0.6\*\*** |
| 置信度（1-7） | 2.1±0.9 | 4.2±1.1 | **5.6±0.8\*\*** |

**\*\* p < 0.01（相比两个基线的配对 t 检验）**

**关键发现：**

- **理解速度**：XAI 系统使理解速度比原始轨迹快 **2.8 倍**，比 LLM 解释快 **1.7 倍**（p < 0.01）。非技术用户提升更显著（vs. 原始轨迹 3.0 倍，vs. LLM 1.7 倍）。
- **准确率**：根因识别准确率显著更高（技术用户 89%，非技术用户 76%），远优于 LLM（68%、52%）和原始轨迹（42%、18%）。
- **修复质量**：我们的 XAI 条件下提出的修复方案评分显著更高（技术用户 4.3/5.0，非技术用户 3.8/5.0），评审员指出建议部分直接促成了更高质量的修复提案。
- **置信度**：用户报告的理解置信度显著更高（技术用户 6.1/7.0，非技术用户 5.6/7.0），这对真实部署中的工具采纳至关重要。

### 5.3 定性发现

事后访谈揭示了若干关键主题：

**视觉上下文至关重要**：20 名参与者中 18 人主动提到执行流程图"极有帮助"或"必不可少"。一位参与者指出："图表立刻让我看到了哪里出了问题。仅凭文字，我得自己拼凑。"

**结构有助于理解**：参与者欣赏解释的一致结构。非技术参与者（主要使用 ChatGPT）指出："ChatGPT 有时聚焦于细节，有时又太宏观。你的系统是一致的。"技术参与者中 7/10 将 Claude 作为默认调试助手，3/10 偏好 ChatGPT。使用 Claude 的参与者提到："Claude 给出非常详细的解释，有时太多了——我只想快速知道什么坏了以及如何修复。"在所有 LLM 偏好中，参与者都更看重我们系统一致的结构化输出，而非临时 LLM 查询的可变质量。

**可操作性最重要**：建议组件被频繁引用为最有价值的功能，对技术用户尤为如此。一位开发者指出："问 Claude 或 ChatGPT 什么错了，他们会告诉我问题，但你的系统给我的是如何修复——带代码示例和配置变更。"非技术参与者也有同感："LLM 解释得很好，但我还是不知道要告诉团队改什么。你的建议是可操作的。"

### 5.4 与通用 LLM 解释的比较

我们的专业化方法相对通用 LLM 的优势：

- **一致性**：我们的系统按一致模板提供结构化输出，而临时 LLM 解释在格式、关注点和完整性上差异显著。
- **视觉整合**：通用 LLM 无法生成执行流程图，而用户研究表明这对理解至关重要。
- **领域专用结构**：我们的系统利用编码智能体故障分类体系提供精确分类和上下文相关建议；通用 LLM 缺乏这种专业结构，每次查询都需从头推断合适类别。
- **可操作指导**：我们的建议引擎提供映射到故障模式的具体、经验证的修复方案；通用 LLM 建议往往更泛化，需要额外解读和验证。
- **自动化潜力**：我们的系统可集成至 CI/CD 流水线实现自动故障分析，而通用 LLM 方法每次故障都需要人工干预。
- **成本效率**：我们的专用系统使用有针对性的提示和结构化输出，相比针对每个故障案例重复临时查询通用 LLM 减少了 token 消耗。

### 5.5 系统局限性

**领域专用性**：当前分类体系和自动分类器专为编码智能体设计，故障类别反映代码生成任务特有的模式。应用于其他领域需要开发该领域的分类体系并重训分类器。

**模型局限性**：由于资源限制，仅使用单一 LLM 提供商（GPT-4.1）进行评估，无法就泛化到其他 LLM 系列（Claude、Gemini、Llama 等）作出声明。

**多并发故障**：当多种故障模式同时出现时（如规划错误和代码生成问题并存），系统有时将故障归因于单一主导类别，可能过度简化根因分析。

**建议上下文依赖**：部分系统级建议（如"增加迭代限制"、"微调模型"）在所有部署上下文中未必可行。

## 6. 讨论

### 6.1 对编码智能体开发的启示

**错误恢复至关重要**：我们数据集中超过一半（56%）的故障源于智能体不尝试从错误中恢复。实现健壮的错误恢复机制——包括提示中的显式错误恢复指令和充足的迭代预算——应成为开发优先事项。

**迭代限制很重要**：许多故障发生在智能体在完成错误恢复前触及迭代限制时。我们的分析表明，编码智能体对于典型问题应有 **5–10 次迭代**的最小预算，复杂任务应有自适应限制。过于严格的限制（1–2 次迭代）即使智能体的初始方法合理也会导致失败。

**从第一天起就重视可观测性**：在开发早期集成可解释性和调试工具能加速开发周期，实现更可靠的智能体系统。事后调试不透明的智能体故障耗时显著更长。

**提示工程很重要**：相当比例的失败（43.8%）是规划失败，其中许多可以通过更好的提示来缓解——包括工作示例、明确的问题分解指令和逐步推理要求。

### 6.2 向编码智能体之外的泛化

整体方法论——开发领域专用故障分类体系、自动分类和结构化解释——可扩展到其他智能体应用：

- **研究智能体**：需要捕捉引用幻觉、文献覆盖不完整或综合错误等故障模式的分类体系。
- **数据分析智能体**：需要统计错误、可视化问题和数据解释失败的分类体系。
- **多智能体系统**：需要协调失败、通信故障和目标冲突的额外故障类别。

每个领域都需要通过系统性分析该领域特有故障来开发专属分类体系。我们的贡献在于证明这类领域专用方法相对通用调试方法具有显著价值。

### 6.3 通用 LLM 在智能体调试中的角色

我们的工作并非表明通用 LLM 不适合智能体调试。我们设想一种互补关系：
- **自动化一线分析**：我们的专用 XAI 系统为常见故障模式提供快速、一致的解释，支持开发期间的快速迭代。
- **深度调查**：对于复杂或新颖故障，开发者可利用通用 LLM 进行探索性分析和假设生成。
- **系统改进**：自动化系统识别的模式为提示、智能体架构和评估框架的改进提供信息。

### 6.4 未来工作

- **跨领域分类体系开发**：系统研究不同智能体领域的故障模式，识别领域专用和通用故障模式。
- **主动学习用于分类体系细化**：结合用户反馈和新发现的故障模式，持续改进分类体系和自动分类器。
- **人在环路精化**：研究开发者对解释的反馈如何改善 XAI 系统和底层智能体。

## 7. 结论

本文提出了一种用于理解和调试基于 LLM 的编码智能体故障的系统性可解释性方法。贡献包括：从 32 个真实故障分析中衍生的综合故障分类体系、达到 82% 准确率的自动分类系统，以及生成可视执行流、自然语言解释和可操作建议的混合解释系统。

用户研究（N=20）证明，相比原始执行轨迹和临时通用 LLM 解释，本方法具有显著改善：故障理解速度提升 **2.8 倍**；根因识别准确率 89% vs. 42%（原始轨迹）；用户置信度显著更高（6.1/7.0 vs. 3.2/7.0）；修复质量显著更高。

重要的是，本工作证明：尽管通用 LLM 能提供有用解释，专业化、领域专用方法提供了可量化的优势——结构一致性、视觉整合、领域专用知识和可操作指导。这些优势的代价是领域专用性，但开发此类专用系统的方法论是可泛化的。

随着基于 LLM 的智能体在软件开发等领域越来越普遍，系统性理解其故障的方法将成为构建可靠、可信 AI 系统的必要条件。我们希望本工作推动一个未来：AI 智能体不仅强大，而且透明、可理解、可调试——使开发者能够系统性地在成功上构建、从失败中学习，而非依赖试错。
