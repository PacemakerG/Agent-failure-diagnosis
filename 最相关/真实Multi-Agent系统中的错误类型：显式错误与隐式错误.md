# 真实 Multi-Agent 系统中的错误类型：显式错误与隐式错误

> 本文档用于评估论文方法的适用性：当阅读新论文时，对照此文档判断该论文解决的是哪类错误，是否与真实 Agent 开发场景匹配。

---

## 一、显式错误（Explicit Failure）

### 定义
错误发生时，系统能自动检测到一个明确的失败信号。

### 典型表现
- 工具调用返回 error（如 Edit 工具写入失败）
- 代码编译不通过、语法报错
- 测试用例运行崩溃、抛出异常
- 进程卡死、超时、无响应

### 特点
- **机器可判断**：有明确的错误节点，程序能自动识别
- **位置明确**：知道在哪一步出了问题
- **容易被现有方法处理**：AgentTrace 等论文针对的就是这类错误

### 在真实 Agent 系统中的实际情况
现代 Multi-Agent 系统几乎不会把显式错误暴露给用户。Agent 遇到工具调用失败会自动重试、换工具；代码跑不通会自动交给 Debug Agent 修复；测试不通过会循环回头修改直到通过。**Agent 系统的自愈机制使得显式错误在最终交付物层面几乎不可见。**

---

## 二、隐式错误（Implicit / Silent Failure）

### 定义
错误发生时，系统没有任何报错信号，每一步从技术上看都"成功"了，但最终交付物不满足预期。

### 典型表现

**状态传递错误：**
- Agent A 完成需求拆解后，应将结果写入 `work_state.md` 的某个字段，但没有写入
- Agent B 基于空白/旧的字段继续执行，它不知道状态缺失
- 后续所有 Agent 都在错误的状态前提下运行
- 最终产物有问题，但没有任何一个节点报错

**测试验证失效：**
- Test Agent 生成的测试用例本身就是错的（Mock 数据是假的、逻辑被简化了、没有覆盖真实边界条件）
- 代码通过了测试，但测试验证的不是真实需求
- 系统认为"任务完成"，实际上交付物是错的

**需求理解偏差：**
- Planner Agent 对需求的理解有偏差，后续所有 Agent 忠实地执行了一个错误的计划
- 每一步执行都是"正确"的，但方向从一开始就错了

**幻觉性完成：**
- Agent 声称完成了某项任务，实际上没有真正执行（如声称更新了文件但没有实际写入）
- 后续 Agent 基于这个虚假的完成信号继续推进

### 特点
- **机器无法判断**：没有错误信号，每个工具调用都返回成功
- **位置不明确**：错误可能发生在任意一步，但没有任何标记
- **只有人能发现**：唯一的错误信号是"用户看了最终产物觉得不对"
- **错误会级联传播**：早期的隐式错误会被后续所有 Agent 放大

## 三、为什么隐式错误才是真实 Agent 系统中的主要问题

### 真实的 Multi-Agent 工作流
```
用户输入需求
    ↓
Planner Agent：拆解需求，更新 work_state.md
    ↓
Coder Agent：基于 work_state 编写代码
    ↓
Test Agent：生成测试用例，运行测试
    ↓（测试不通过 → 自动循环修复）
Review Agent：审查代码
    ↓
交付产物
```

在这个流程里：
- 工具调用失败会被自动重试和修复
- 编译错误会被自动修复直到通过
- 测试不通过会被自动循环修复直到通过

**最终一定会有一个"技术上成功"的交付产物。**

真正的问题是：这个技术上成功的产物，是不是用户真正想要的？

### 隐式错误的根本困难

**谁来定义"正确"？**
- 编译通过 → 机器能判断 ✓
- 测试通过 → 机器能判断，但测试本身可能是错的 ✗
- 需求满足 → 只有人能判断，或需要极其精确的需求规格 ✗

没有客观的、机器可判断的"正确"标准，就无法自动识别隐式错误，也就无法定位根因。

## 四、用此文档评估论文的方法

阅读新论文时，提问以下几个问题：

**Q1：这篇论文假设存在一个明确的错误节点（`v_error`）吗？**
- 是 → 它针对的是显式错误，在真实场景中适用性有限

**Q2：它的测试数据集是合成的（人为注入 bug）还是真实的（有机故障）？**
- 合成注入 → 控制了错误类型和位置，和真实场景差距大
- 有机故障 → 更接近真实，但仍需看是否包含隐式错误

**Q3：它的"错误信号"是什么？**
- 程序报错/编译失败 → 显式错误范畴
- 测试不通过 → 半显式，取决于测试质量
- 最终输出不满足需求 → 隐式错误范畴，看论文如何处理这个判断

**Q4：它能处理"Agent 自愈后仍然错误"的场景吗？**
- 即：所有工具调用都成功，测试也通过，但结果是错的

## 五、目前的开放问题

隐式错误的诊断目前没有好的解决方案，核心挑战：

1. **错误检测问题**：如何自动识别"结果是错的"
2. **错误定位问题**：在没有报错节点的情况下，如何确定因果链的起点
3. **验证有效性问题**：如何判断 Agent 的自我验证（测试）本身是否有效
4. **状态一致性问题**：如何追踪多 Agent 间的状态传递是否正确

这四个问题都是当前学术界的开放问题，现有论文（AgentTrace、ErrorProbe、XAI）均未真正解决。
